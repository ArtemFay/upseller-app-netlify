# Подбор — контекст разработки

> **Источник правды для кода**: `web/podbor/` (frontend) + `web/api/podbor-*.js` (backend) + `web/api/_lib/podbor/` (общая логика). Этот файл — только бизнес-документация и UX-договорённости.

Этот файл — оперативная память проекта: бизнес-правила подбора, UX-договорённости, контракт API, статус. Обновляется в каждой значимой сессии.

Корневой контекст по фулфилменту см. в `1_CONST/`. Здесь — только то, что специфично для **этого web-приложения** (планшетный подбор), плюс расширения, которых в CONST ещё нет.

## 1. Назначение приложения

Промежуточная (web/планшет) реализация процесса **«ручной подбор по таблице»** — заменяет работу подборщика в Google Sheets-таблицах «Планшет 1 / Планшет 2 / …». Архитектурное требование: тот же бизнес-процесс должен работать **на четырёх устройствах** — десктоп 4:3 (≥ 1280 px, базово редкое использование), планшет (~1024 px), телефон (≤ 600 px), ТСД (≤ 480 px + сканер). Один бэкенд, одна БД, разные UI-обёртки. См. § 6.

## 2. Источники данных

**Главный источник стартового экрана — `ПОДБОРЫ.БД_ЭКСП`** (16 колонок A:P, см. полную схему в `1_CONST/03_CURRENT_GAS_SYSTEM.md` § 5). В одном запросе получаем **всё**, что нужно для стартового экрана **любого** устройства: список заявок, клиентов, статусов, направлений, потребности по баркодам.

| Что | Откуда сейчас | Куда планируется |
|---|---|---|
| Список **всех активных заявок** (10–30 шт) и клиентов | `ПОДБОРЫ.БД_ЭКСП!A2:P` (один запрос на старте) | БД WMS |
| **Карточка заявки** (`КС`, `СКЮ`, `ЕД`, `СТАТУС`, `МП`, `СКЛАД`, `КОНЕЧН СКЛАД`, дата отгр) | `БД_ЭКСП` колонки B/F/G/H/M/N/O | БД WMS |
| **Структура заявки** — нужные баркоды и количества (`НУЖН`) | `БД_ЭКСП.J` (`ЛОГ ЗАЯВКИ`, см. § 3.4 — формат с U+2060) | БД WMS |
| Короба клиента (содержимое полотна) | `UPSELLER!🍬 КОРОБЫ!C7:U` (см. `lib/podbory-load.js`) | БД WMS |
| Состояние раскладки (галочки, КОЛ ПОДБ/ПЕРЕМ, КУДА…) | event-store JSON `${PODBOR_DATA_DIR}/zayavki/<id>.json` (events + computed) | БД WMS |
| Открытые коробы отгрузки активной заявки | `🍬 КОРОБЫ` где `F = <zayavkaId>` и статус `В СБОРКЕ` | БД WMS |
| `ЯЧ`-ячейки клиента | `🍬 КОРОБЫ` где `B = ЯЧ AND T = client` | БД WMS |
| **Начисления** (зарплата подборщикам) | **Отдельная таблица `НАЧИСЛЕНИЯ`** (`NACHISLENIYA_ID=1tbUsKXEZK_…`), лист `НАЧ`, A:O 15 колонок (см. nach-writer.js). НЕ путать с ПОДБОРЫ. | БД WMS |
| **finish-summary** (статус СОБРАНО + сводка по заявке) | `ПОДБОРЫ.БД` колонки `U` (статус), `V` (timestamp), `W:AJ` (24 поля: SKU, ед, free/paid, коробы, сборы, лог, длительности) | БД WMS |
| **Архив завершённых заявок** | JSON-снимки в `${PODBOR_DATA_DIR}/_done/<id>-<ISO_ts>.json` (после успешного finish), доступны через UI `/podbor/archive.html` | БД WMS |

SA-доступ ко всем таблицам уже есть.

## 3. Структура источников

### 3.1. Прототип полотна — лист `ПОДБОР` Планшета

Полная схема — в `1_CONST/03_CURRENT_GAS_SYSTEM.md` § 5a (29 колонок A:AD). Конспект для UX:

- **Чтение** (read-only): `ТИП / SKU / МП / КЛИЕНТ / ВСЕГО В КОР / СПИС ЯЧЕЕК / ТАРА / СТАТУС / КОРОБ / БАРКОД / БАР5 / АДР / КОЛ`.
- **Редактирование подборщиком**:
  - **Раскладка**: `КОЛ ПОДБ / КУДА ПОДБ` + `КОЛ ПЕРЕМ / КУДА ПЕРЕМ` + `ОСТ` (derived) + `ПОЛН КОРОБ` (флаг).
  - **Микро-инвент**: `НОВ КОЛ`, `НОВ АДР`.
- **Подсказки агрегата**: `НУЖН / СОБР / ЕЩЕ` per BAR5 для активной заявки.

### 3.2. Реестр заявок — лист `БД_ЭКСП`

Полная схема — в `1_CONST/03_CURRENT_GAS_SYSTEM.md` § 5 (16 колонок A:P). В этом листе **все открытые** заявки (`СОЗДАНО / В РАБОТЕ / ЧАСТИЧНО СОБРАНА`) — стартовая загрузка приложения.

### 3.3. Унифицированный ключ заявки — только как подпись, не навигация

Изначально планировался для dropdown'а заявок, но **отвергнут** как основная навигация: при 100+ активных заявках сжатый ключ всё равно сложно сканировать глазом. Вместо dropdown — **список карточек** (см. § 5.0).

Формат может использоваться **локально** как одно-строчная подпись внутри карточки на ТСД (где места очень мало) или в логах:

```
<C>_<client_5>_<E>_<F_2>_<H_5>
```

`C`=`№ ПОДБОРА`, `client_5`=первые 5 значимых символов клиента, `E`=дата отгр (`DD.MM`), `F_2`=МП всегда 2 символа (`НЕТ`→`НЕ`), `H_5`=первые 5 символов конечного склада. Пример: `S1270-ВИДИНЕЕВА_ВИДИН_25.04_НЕ_Блоге`.

### 3.4. Парсер `ЛОГ ЗАЯВКИ` (колонка J `БД_ЭКСП`)

Содержимое `J` — структура заявки в одной ячейке:

```
<barcode>⁠ - ⁠<qty>⁠
<barcode>⁠ - ⁠<qty>⁠
...
```

Между баркодом, тире и количеством — **U+2060 (WORD JOINER, невидимый)**. Между записями — `\n`. Устойчивый парсер:

```js
function parseLogZayavki(j) {
  return String(j || '')
    .split('\n')
    .map(line => line.match(/\d+/g))     // первые два числа в строке
    .filter(m => m && m.length >= 2)
    .map(([barcode, qty]) => ({ barcode, qty: Number(qty) }));
}
```

Не сплитить по ` - ` — невидимый U+2060 ломает наивный split. См. CONST/03 § 5.

## 4. Бизнес-инварианты подбора

1. **Короб-источник = одна строка полотна на каждый баркод-в-коробе.** Один физический короб может занимать несколько строк, если в нём несколько баркодов. Для модалки и тарификации именно **физический короб** — атомарная единица (см. § 7).
2. **Раскладка строки источника** идёт максимум по трём потокам, инвариант:
   `КОЛ ПОДБ + КОЛ ПЕРЕМ + ОСТ = КОЛ` (или `НОВ КОЛ`, если был микро-инвент).
   - `КОЛ ПОДБ → КУДА ПОДБ`: в **короб отгрузки** (закрытая тара, любая нормальная).
   - `КОЛ ПЕРЕМ → КУДА ПЕРЕМ`: в **ячейку хранения** (тара `ЯЧ`, открытый короб).
   - `ОСТ`: остаётся в источнике.
   - Любая компонента может быть нулевой.
3. **Жёсткий инвариант изъятия**: `КОЛ ПОДБ + КОЛ ПЕРЕМ ≤ КОЛ` (или ≤ `НОВ КОЛ`). Превышение → блок UI с подсказкой «нужен микро-инвент» (нельзя изъять больше, чем фактически в коробе).
4. **`ПОЛН КОРОБ`** (флаг физического короба, не строки): «беру короб целиком на отгрузку». Распространяется **на все строки этого короба**: для каждой `КОЛ ПОДБ = КОЛ`, `КОЛ ПЕРЕМ = 0`, `ОСТ = 0`, единый `КУДА ПОДБ` для всего короба. Доступен **только** если **каждый** баркод в коробе требуется активной заявкой в кол-ве ≥ `КОЛ` для своей строки. Если хоть один баркод не нужен (или нужен меньше) — флаг отключён, в tooltip объясняется причина. Обходной путь — обнулить «лишний» баркод через микро-инвент, если это расхождение с фактом.
5. **`НУЖН / СОБР / ЕЩЕ`** — **на уровне баркода** для активной заявки. В UI показываем в шапке группы и в модалке, не дублируя в каждой строке.
6. **Микро-инвент (`НОВ КОЛ`) — НЕ обязателен по умолчанию.** По умолчанию подразумеваем, что системные данные верны. Микро-инвент инициирует **сам подборщик**, когда заметил расхождение. Триггеры обязательного микро-инвента:
   - попытка ввести `КОЛ ПОДБ + КОЛ ПЕРЕМ > КОЛ` (нельзя изъять больше);
   - попытка поставить `ПОЛН КОРОБ` в коробе с не-заявочным баркодом, который по факту отсутствует (тогда `НОВ КОЛ = 0` снимает блокировку).
   После применения микро-инвента в БД обновляется `КОЛ` строки, далее раскладка ведётся от новых данных. Параллельно работающие подборщики увидят свежее состояние при следующем `box.open`.
7. **Активный фильтр заявки**: после выбора заявки скрываются баркоды с `НУЖН = 0` для этой заявки (по умолчанию). При открытии модалки конкретного короба показываем **все** баркоды в нём — потому что подборщик может перекладывать в ячейки и не-заявочный товар (освобождая короб).

## 5. UX-договорённости (текущее состояние)

### 5.0. Авторизация и стартовый экран

**Авторизация**: Google OAuth через белый список email-ов (как в INVENT_WEB на Netlify). Только одобренные пользователи могут войти. После входа в `req.user` (бэк) и в JS-стейте (фронт) есть `{ email, name, surname, picture, role }`.

**Идентификатор пользователя на экране** (всегда виден в правом углу топбара):
- Формат: `Имя Ф.` — имя целиком + первая буква фамилии заглавной с точкой. Пример: «Артём Файзулов» → `Артём Ф.`
- Альтернативный режим (по умолчанию на десктопе/планшете) — фото из Google + полное имя.
- На ТСД (узкий экран) — только инициалы `АФ` (первые буквы имени и фамилии).
- Клик по идентификатору → выпадающее меню: «Выйти», «Сменить аккаунт».

**Роли в приложении** (определяются по email через серверный whitelist):
- `picker` (рядовой подборщик): может работать со **своими** заявками (где `lockedBy = self`). Не может менять статусы чужих заявок.
- `senior` (старший приёмщик / старший упаковщик / **кладовщик** — все имеют админ-права на заявки подбора, см. `02_BUSINESS_PROCESSES.md` §1): может снимать lock с чужой заявки, менять её статус, завершать.
- `admin` (системный админ): всё что senior + редактирование whitelist-а пользователей (это уже в админ-панели, не в подборе).

При открытии приложения делается один запрос `/api/zayavki-list` → читает весь `БД_ЭКСП`, парсит `J`, возвращает массив заявок:

```ts
type Zayavka = {
  number: string;        // C, "S1270-ВИДИНЕЕВА"
  client: string;        // A
  ks: number;            // B, default 1
  dateZay: string;       // D, "DD.MM"
  dateOtgr: string;      // E
  mp: string;            // F, 2 chars or "НЕТ"
  warehouse: string;     // G
  finalWarehouse: string;// H
  comment: string;       // I
  items: { barcode: string; qty: number }[]; // J разобран (потребность)
  skuCount: number;      // M
  unitsTotal: number;    // N
  status: 'СОЗДАНО' | 'В РАБОТЕ' | 'ЧАСТИЧНО СОБРАНА';
  type: 'ОТГ' | 'ПЕР';   // P — тип заявки (отгрузка / перемаркировка). Пусто → ОТГ.
  pickMode: 'СВОБ' | 'КОР' | 'КОР+';
                         // Q — режим сборки. Пусто → СВОБ.
                         // СВОБ — свободно (поштучно из любых коробов, можно миксовать).
                         // КОР  — только полные короба, нельзя дербанить и миксовать.
                         //         Микро-инвент разрешён (предсборочный пересчёт).
                         // КОР+ — приоритет полные короба; раздербанить разрешено
                         //         только ОДИН для остатка. См. TSD_STANDARD.md § 1a.
  lockedBy?: string;     // если статус = 'В РАБОТЕ', имя/id оператора
};
```

UI стартового экрана:

1. **Селектор клиента** (вверху, всегда виден):
   - Опционален. По умолчанию — «Все клиенты».
   - Выбор клиента → (а) фильтрует список карточек ниже до этого клиента, (б) на **десктопе/планшете** — стартует **фоновую** загрузку коробов клиента (`/api/load?client=…`) пока подборщик читает карточки. На **телефоне/ТСД** не префетчим (память/трафик).
2. **Список карточек заявок** (Vertical scroll, не dropdown):
   - Каждая карточка — одна заявка из `БД_ЭКСП`. На карточке видно сразу:
     - № заявки крупно;
     - Статус-бейдж: `СОЗДАНО` (серый), `ЧАСТИЧНО СОБРАНА` (жёлтый), `В РАБОТЕ` (синий, c подписью «работает: <имя>»);
     - Клиент · `КС` (если ≠ 1 — выделить «×0.6» или «×2»);
     - Дата отгрузки · МП · Склад → Конечный склад (одной строкой);
     - `СКЮ: M · Ед: N`;
     - **Кнопка `[Начать]`** — активна только если статус ∈ {`СОЗДАНО`, `ЧАСТИЧНО СОБРАНА`}. Для `В РАБОТЕ` — серая кнопка `[Занято]` с tooltip «работает: <имя> с <время>».
   - Сортировка: сначала `ЧАСТИЧНО СОБРАНА` (продолжаем своё), потом `СОЗДАНО` по дате отгрузки ↑, в конце `В РАБОТЕ` (информационно).
3. **Клик `[Начать]`**:
   - Атом `zayavka.start` → бэк меняет `O = В РАБОТЕ`, ставит `lockedBy = currentUser`, возвращает заявку.
   - Если в фоне коробы клиента уже подгружены (см. п.1) — переход на полотно мгновенный. Иначе — спинер «загружаем коробы…» 1–3 сек.
   - Полотно показывается с активным фильтром по `items` заявки (см. § 5.9).
4. **Шапка с заявкой** на полотне (sticky, всегда видна):
   - № заявки · статус · `СКЮ: M / X собрано · Ед: N / Y собрано`;
   - Кнопки: `[Структура заявки]` (открывает Zayavka-modal, см. § 5.0a), `[Частичный отчёт]` (status → `ЧАСТИЧНО СОБРАНА`, можно вернуться позже), `[Завершить]` (закрытие подбора).
5. **Без активной заявки** полотно не показывается. Сразу после старта — стартовый экран.

Адаптация:
- **Десктоп 4:3 / планшет**: список карточек в две колонки + клиент-селектор слева. Префетч коробов на выбор клиента.
- **Телефон**: один столбец карточек, клиент-селектор сверху. Без префетча — коробы грузятся только при `Начать`.
- **ТСД**: один столбец, крупные кнопки `Начать` (легко нажать пальцем). Альтернатива выбора — **скан QR-кода заявки** на печатном листе (если печатают) → автоматический `Начать`.

### 5.0a. Модалка структуры заявки (Zayavka-modal)

Открывается из шапки полотна по `[Структура заявки]`. Таблица баркодов заявки с **пятью** столбцами прогресса:

| Баркод (BAR5 + полный) | SKU (название) | Заявлено | Доступно | Собрано | Ещё |
|---|---|---|---|---|---|
| 05231 / 2049896213734 | Спорт. костюм 44 | 50 | 80 | 15 | 35 |
| 28092 / 2049897028092 | Платье · 46 | 100 | 60 | 60 | 0 |
| … | … | … | … | … | … |

- **Заявлено** (`request`) — из `items[].qty` заявки (`J` распарсенный).
- **Доступно** (`available_for_picking`) — расчёт из `🍬 КОРОБЫ`: сумма `КОЛ` по строкам где `T = client AND U = barcode AND статус ∈ {ГОТОВО, ХРАНЕНИЕ} AND тип ≠ 'БРАК'`. Подробно — `1_CONST/03_CURRENT_GAS_SYSTEM.md` § 3. Брак исключаем всегда. **Доступно может быть и больше, и меньше Заявленного** — это разные оси (склад vs заявка).
- **Собрано** (`picked`) — агрегат: `Σ КОЛ ПОДБ` по всем строкам полотна с этим баркодом и активной заявкой. Считается на фронте из локального стейта.
- **Ещё** (`still_needed`) = `min(Заявлено, Доступно) − Собрано`, но не меньше 0. Это «сколько ещё реально можно собрать». Если `Доступно < Заявлено`, реалистичная цель — `Доступно`, не `Заявлено`.
- **Подсветка**:
  - `Доступно < Заявлено` → ячейка `Доступно` оранжевая (на складе меньше, чем заявлено — клиент получит недосбор);
  - `Доступно = 0` → строка красная (товар отсутствует на складе);
  - `Собрано = min(Заявлено, Доступно)` → строка зелёная (всё, что можно — собрано);
  - `Собрано > Доступно` → не должно происходить, но если получилось → красное предупреждение (баг расчёта `Доступно` или работа без обновления данных).
- SKU — джойнится из `🍬 КОРОБЫ` (любая строка с этим баркодом). Если ни одной — поле пустое, баркод выделяется красным как «нет на складе».
- Сортировка: сначала строки с `Ещё > 0` (текущая работа), потом `Ещё = 0` (закрыто).

В **шапке группы полотна** (`🏷️ BAR5: 05231`) дублируем компактно: `Заявлено: 50 · Доступно: 80 · Собрано: 15 · Ещё: 35` (на телефоне сворачивается до `50/80/15/35`). Это чтобы не открывать модалку для каждого баркода — основные числа сразу видны.

### 5.0b. Завершение и закрытие заявки

Подбор заявки можно завершить (`Завершить`) или закрыть без подбора (`Закрыть`). Это **два разных действия** с разной семантикой:

| Действие | Когда доступно | Что делает | Тарификация |
|---|---|---|---|
| **`Закрыть`** | Если `Σ Собрано = 0` (не подбирали ни одной единицы) | Снимает lock, статус → `СОЗДАНО` (заявка возвращается к ожиданию). Никаких `PickEvent`. | `0 ₽` |
| **`Частичный отчёт`** | В любой момент после `Σ Собрано > 0` | Снимает lock, статус → `ЧАСТИЧНО СОБРАНА`. Можно вернуться позже (любой подборщик с правом, см. § 5.0). `PickEvent` не закрываются. | не списываем |
| **`Завершить`** | См. условия ниже | Финализация: расчёт `PickEvent` per box (free/paid, см. § 7), `FinancialTransaction`, запись в `ВР`, заявка пропадает из `БД_ЭКСП` | списание ×`КС` |

**Условия `Завершить`**:

- `Σ Собрано > 0` (нельзя завершать с нулём — для этого `Закрыть`).
- Для каждого баркода заявки **должно** выполняться `Собрано ≥ min(Заявлено, Доступно)`. То есть либо подобрано всё что заявлено, либо подобрано всё что физически было на складе.
- Если **по какому-то баркоду подобрано меньше**, чем доступно (на складе ещё есть, а подборщик не нашёл) → блок завершения. Чтобы разблокировать:
  1. **Микро-инвент** оставшихся коробов с этим баркодом (`НОВ КОЛ = 0` для тех, где подборщик не нашёл). После этого `Доступно` пересчитывается → может стать = `Собрано` → можно завершать.
  2. ИЛИ **комментарий-причина** (если подборщик не имеет права/возможности обнулять короба, например, не может найти физически — короб может быть в другой смене перепрятан). В этом кейсе после `Завершить` создаётся отдельный таск для кладовщика «уточнить наличие баркодов X, Y».
- Если `Заявлено > Доступно` и `Собрано = Доступно` (взяли всё, что было) → завершение **без** микро-инвента и **без** комментария. Это нормальный кейс «склад не успел дозаказать», сообщение клиенту получит логист отдельно.
- **Комментарий причины недосбора** обязателен только в кейсе «`Собрано < Доступно AND Собрано < Заявлено`». Текстовое поле + опциональная аудио-запись (для не-русскоязычных сотрудников). Хранится в `AuditLog.details.partial_finish_reason`.

**Алгоритм UI при попытке `Завершить`**:

```
1. Считаем для каждого баркода заявки: shortage = min(Заявлено, Доступно) - Собрано
2. Если все shortage = 0 → завершить молча.
3. Если есть baрcodes с shortage > 0:
   3a. Открыть диалог «Не всё собрано» со списком проблемных баркодов
       и для каждого — кнопкой «Микро-инвент оставшихся коробов» (открывает BoxModal-ы по одному).
   3b. Внизу диалога — поле «Причина недосбора» (текст + 🎤).
   3c. Кнопка `Завершить` активна только если: либо все shortage устранены через
       микро-инвент (Доступно подкрутилось вниз), либо причина заполнена.
4. На подтверждение → POST /api/zayavka.finish с { reason?: string, partialBarcodes: [...] }.
```

При завершении заявки списание считается:

```
total_charge = Σ_box charge(box)        # см. § 7 (free/paid)
total_charge *= zayavka.КС               # коэффициент сложности из БД_ЭКСП.B
```

### 5.1. Полотно — read-only, modal-driven

Полотно показывает группы по `BAR5` в одной таблице с `table-layout: fixed`. **В строке нет редактируемых элементов** — никаких inline-чекбоксов и numpad'ов. Любое действие — через **модалку короба** (§ 5.6).

Колонки полотна (планшет, ≥ 1024 px):
- SYNC · Адрес · Другие ячейки · **Короб** · Тара · Статус · Тип · Кол · Все · Sku-в-коробе · _badge раскладки_

Скрыты по умолчанию (вынесены в Настройки, § 5.8): `SKU` (артикул), `МП`, `БАРКОД` (полный — он уже виден крупно в шапке группы по BAR5).

### 5.2. Соотношение сторон

`body { max-width: calc(100vh * 4 / 3) }` — на ультра-широких мониторах контент ≤ 4:3, центрирован.

### 5.3. Загрузка данных — двухступенчатая, с условным префетчем

1. **При запуске приложения** → один запрос на `БД_ЭКСП` (см. § 5.0). Получаем все заявки с распарсенным `J`. Этого достаточно для стартового экрана.
2. **Префетч коробов клиента** (только на десктопе/планшете): срабатывает при выборе клиента в селекторе стартового экрана. Запрос `/api/load?client=…` уходит в фон, пока подборщик ещё читает карточки заявок. Кеш `boxesCacheByClient` живёт пока сессия активна.
3. **На клик `[Начать]`**: если кеш для клиента есть — переход на полотно мгновенный. Если нет (телефон/ТСД, или префетч ещё в полёте) — спинер «загружаем коробы…» и ожидание. После загрузки полотно фильтруется по `items` заявки (`НУЖН > 0`, см. § 5.9).

Двухступенчатость даёт:
- быстрый старт (`БД_ЭКСП` ~10–30 строк, <1 сек);
- мгновенный переход на полотно на планшете/десктопе (фоновая подгрузка);
- экономию трафика на телефоне/ТСД (грузим только когда нажали `Начать`);
- возможность **переключать заявки одного клиента** без перезагрузки полотна;
- микро-инвент случайных коробов (не из заявки), пока полотно подгружено.

### 5.4. Пагинация

Soft-target ~50 строк/стр., толеранс **35..65**. **Bar5-группа никогда не разрезается** между страницами. Пагинатор sticky.

### 5.5. Синхронизация

`SyncQueue` (debounce 400 мс) шлёт батчи в `POST /api/sync`. Индикатор статуса — первая колонка строки и общий статус сверху. Семантика: `idle / connected / syncing / synced / synced+changed / error`.

### 5.6. Модалка короба (BoxModal) ← ключевое решение

Открывается **по тапу на любую строку группы или на номер короба** (а на ТСД — по сканированию QR/баркода короба). **Все** действия раскладки/инвента/изменения адреса делаются здесь.

Структура модалки (сверху вниз):

1. **Шапка короба**:
   - Номер короба, статус, тара, текущий адрес.
   - Бейдж тарификации (динамический): `🆓 Бесплатный` / `💰 Штучный · N шт` (см. § 7).
   - Кнопки шапки: `Изменить адрес`, `Полный короб → отгрузка` (доступно только если все строки имеют `ПОДБ ≥ 1` и `ПЕРЕМ = 0`; иначе тап показывает почему недоступно).

2. **Список баркодов в коробе** (1..N карточек, по одной на каждый баркод):
   - BAR5 + полный баркод + краткое имя SKU.
   - Текущее кол-во в коробе (`КОЛ`), с inline-кнопкой `[✏ микро-инвент]` (§ 5.10).
   - Подсказка `НУЖН / СОБР / ЕЩЕ` для активной заявки (если этот баркод участвует в заявке).
   - Три слота раскладки:
     - `📦 → отгрузка`: autocomplete по открытым коробам отгрузки (или новый), numpad для штук. Если баркод в заявке и `ЕЩЕ > 0` → numpad по умолчанию заполнен `min(КОЛ, ЕЩЕ)`.
     - `🗄️ → ячейка`: autocomplete по `ЯЧ`-коробам клиента, numpad.
     - `Остаётся в источнике`: derived, нередактируемо: `КОЛ − ПОДБ − ПЕРЕМ`.

3. **Подвал**:
   - Сводка по коробу: `всего изъято = Σ КОЛ ПОДБ`, `всего в ячейки = Σ КОЛ ПЕРЕМ`, `остаток = Σ ОСТ`.
   - Кнопки `[Сохранить]` (триггерит `SyncQueue`) и `[Отмена]`.

**Принципы модалки**:
- На планшете — центрированная full-screen в landscape (~80% ширины × 90% высоты).
- На телефоне — bottom-sheet на 100% высоты (стандартный мобильный паттерн).
- На ТСД — full-screen, сканер автоматически кидает текст в активный input (если активен слот «отгрузка/ячейка»), иначе скан переключает в режим выбора (`КОРОБ:NNN` → подставляет в КУДА ПОДБ; `Я:NNN` → в КУДА ПЕРЕМ; голый barcode → джамп к карточке этого баркода в открытой модалке).
- Чекбокс `ПОЛН КОРОБ` **спрятан только в модалке** — нельзя кликнуть, не увидев полное содержимое короба. Это защита от ошибки (в коробе может быть несколько баркодов, и «полный» = «забираю все из короба», а не из одной строки).

### 5.7. Создание коробов отгрузки (батчем) + предварительная печать этикеток

Кнопка `[+ Короба отгрузки]` в топбаре рабочего пространства (после выбора заявки). По клику — мини-модалка:

- **Сколько создать?** numpad: 1, 3, 5, 10, … (типичная стратегия — создать с запасом, например 30 коробов на заявку 300 ед × 10 шт/короб).
- **Тара по умолчанию**: select (`К_1.0` обычно), можно переопределить позже на коробе.
- ☐ **Сразу распечатать этикетки** (по умолчанию включено).

При подтверждении система выделяет N следующих свободных номеров под текущей заявкой и создаёт их со статусом «открыт». **Схема нумерации** (см. CONST §03 §12):
- Префикс = id заявки на отгрузку. Заявка `S1294-Позаян` → префикс `S1294` (`R<NNNN>` для возвратных).
- Полное имя короба: `S<NNNN>-<MMM>` где MMM — порядковый номер с лидирующими нулями: `S1294-001`, `S1294-002`, …

В UI (autocomplete `КУДА ПОДБ`, кнопки выбора в модалке) подборщик видит и оперирует **коротким порядковым номером** (`1, 2, 3, …`); система автоматически разворачивает в полное имя при сохранении в БД и в Sheets.

Создание обязательно **до** первой раскладки (нельзя «положить в отгрузку», не имея получателя). Если подборщик попробует разложить без созданных коробов — модалка короба-источника предложит создать пачку на лету (`[+ создать N коробов]`).

#### 5.7a. Этикетки коробов отгрузки (термопринтер)

После создания коробов система формирует **ленту из N этикеток** на термопринтере (типовая лента 58×40 мм, как для товарных этикеток у оператора, см. CONST/02 §1 OPERATOR). Этикетки печатаются **подряд одной лентой** — подборщик отрывает по одной и наклеивает на физические коробы.

**Содержимое этикетки** (одна на короб):

```text
┌─────────────────────────────┐
│   ┌─────┐                   │
│   │ QR  │   №  001          │  ← крупно: порядковый номер (видно издалека)
│   │     │                   │
│   └─────┘   S1294-001       │  ← полное имя короба мелким шрифтом
│             ВИДИНЕЕВА        │  ← клиент (для контроля)
│             25.04 · НЕТ     │  ← дата отгрузки · МП
└─────────────────────────────┘
```

QR-код кодирует **полное имя короба** (`S1294-001`) — лучше как plain text, не URL. ТСД сканирует → получает имя короба → идентифицирует строку в БД.

**Зачем этикетки**:

1. **Визуальная навигация** — подборщик с одного взгляда понимает «короб 5» vs «короб 6» в куче пустых коробов; не путается.
2. **Параллельная работа** — раздав этикетки между двумя-тремя подборщиками одной заявки, можно физически разделить полку: «ты собираешь в коробы 1–10, я — в 11–20». В UI каждый выбирает свой подмножество (через autocomplete или скан) и независимо раскладывает (см. §11 про конкурентность).
3. **Скан на ТСД вместо набора номера** — главный аргумент для целевого устройства.

**Печать**:

- На этапе MVP (этап 2 разработки) — кнопка `[Печать ленты]` в мини-модалке; backend генерирует PDF на 58×40, фронт отправляет на печать через `window.print()` или Web Bluetooth (для Brother/Godex).
- Конкретный термопринтер — TBD (нужно подтвердить с пользователем модель). Для совместимости лучше PDF/ZPL, чтобы работало с любой моделью на офисном Wi-Fi.
- Допустимо допечатать ленту повторно (например, потеряли часть). Это не создаёт новых коробов — только переиздаёт этикетки уже существующих.

**Альтернативный путь без печати** (для дев-режима, телефона без термопринтера, или ТСД на удалении):

- Подборщик создаёт коробы в UI без печати, но тогда обязан сам надписать номер маркером на коробе. Менее удобно, но работоспособно.

### 5.8. Настройки (gear icon)

Иконка ⚙️ в топбаре → drawer/dialog с настройками сессии:
- ☐ Показывать колонку **МП** (по умолчанию выключено).
- ☐ Показывать **полный баркод** в строке (по умолчанию выключено — он есть в шапке группы по BAR5).
- ☐ Показывать колонку **SKU** (артикул) — по умолчанию выключено для планшета, включено только если очень широкий экран.
- ☐ Показывать **все коробы**, не только участвующие в активной заявке (по умолчанию выключено — после выбора заявки фильтруем по `НУЖН > 0`).
- ☐ Компактный режим (телефон) — карточки вместо таблицы.

Состояние настроек хранится в `localStorage`, привязано к user-id (когда будет авторизация).

### 5.9. Активный фильтр заявки

- Клиент выбран → загружаем **все** короба → отображаем все.
- Заявка выбрана → скрываем баркоды, у которых `НУЖН = 0` для этой заявки. **Группы (bar5)** скрываются целиком (если ни в одной строке группы баркод не участвует в заявке).
- Внутри открытой модалки короба фильтр **не действует** — там виден весь физический состав короба.
- Сброс заявки → снова показываем всё.

### 5.10. Микро-инвент

Микро-инвент **не обязателен по умолчанию** — система доверяет своим данным. Подборщик инициирует его сам, заметив расхождение (или система принуждает в двух кейсах: при превышении изъятия и при попытке `ПОЛН КОРОБ` для не-заявочного баркода).

Кнопка `[✏ микро-инвент]` рядом с `КОЛ` в карточке баркода в модалке. По клику:
- Open numpad для нового количества (`НОВ КОЛ`, может быть и `0`).
- При сохранении — сразу `POST /api/sync` атомом `box.inventory_correction` (см. § 6.1). Применяется в БД до возврата.
- После применения: `КОЛ` строки = `НОВ КОЛ`, производные пересчитываются, кнопки раскладки разблокируются.
- Изменение пишется в `AuditLog` (operation = `inventory_correction`, причина = свободный текст, опционально).

**Запрет**: пока есть **непримёненный** микро-инвент в карточке (поле редактируется, но не сохранено), нельзя править `КОЛ ПОДБ / ПЕРЕМ` для этого баркода. Логика: «сначала зафиксируй факт в системе — потом разделяй». Параллельно работающий другой подборщик в свою следующую загрузку получит свежее `КОЛ`.

## 6. Атомарная модель действий (multi-device)

Чтобы десктоп / планшет / телефон / ТСД работали через одну и ту же бизнес-логику, мы выделяем **атомы операций**. Каждый атом = одна команда к бэкенду (`POST /api/...`), идемпотентный, с уникальным client-side-id для retry-логики `SyncQueue`.

### 6.1. Атомы

| Атом | Контракт | Когда |
|---|---|---|
| `auth.me` | `{}` → `{ email, name, surname, picture, role }` | при загрузке приложения — кто я. На сервере определяется по сессии Google OAuth + whitelist. |
| `zayavki.list` | `{}` → `Zayavka[]` (см. § 5.0) | старт приложения; обновление списка при возврате на стартовый экран |
| `zayavka.start` | `{ zayavkaNumber }` → `{ zayavka, lockToken }` | клик `[Начать]` — статус → `В РАБОТЕ`, ставится lock c `lockedBy = currentUser.email`, `lockedAt = now`. Идемпотентен для **того же** пользователя (повторный старт = refresh). Для **другого** — 409 Conflict (`role != senior`) или принудительный override (`role ∈ {senior, admin}`). |
| `zayavka.unlock` | `{ zayavkaNumber }` → `{ zayavka }` | senior/admin снимает lock с чужой заявки. Статус → `ЧАСТИЧНО СОБРАНА` (если было `Собрано > 0`) или → `СОЗДАНО`. |
| `zayavka.partial_close` | `{ zayavkaNumber, lockToken }` → `{ zayavka }` | кнопка «Частичный отчёт» — статус → `ЧАСТИЧНО СОБРАНА`, lock снимается. Можно вернуться позже (тот же подборщик или другой с правом). |
| `zayavka.close` | `{ zayavkaNumber, lockToken }` → `{ zayavka }` | кнопка «Закрыть» — допустима только если `Σ Собрано = 0`. Снимает lock, статус → `СОЗДАНО`. Никаких `PickEvent` не создаётся. |
| `zayavka.finish` | `{ zayavkaNumber, lockToken, reason?: string, audioBlob?: blob }` → `{ pickEvents, totalCharge }` | «Завершить» — финализация: расчёт `PickEvent` per box (free/paid, см. § 7) × `КС`, запись в `ВР` и `НАЧИСЛЕНИЯ`, заявка пропадает из `БД_ЭКСП`. `reason` обязателен только если `Собрано < min(Заявлено, Доступно)` для какого-то баркода (см. § 5.0b). |
| `client.load` | `{ client }` → `{ groups: Group[], availability: { [barcode]: number } }` | нажатие «Загрузить полотно» или фоновый префетч. `availability` — посчитанное на бэке `available_for_picking` per баркод (см. CONST/03 § 3). |
| `box.open` | `{ boxId }` → `{ rows: BoxRow[], shipmentBoxes: ShipBox[], cells: Cell[] }` | подборщик открывает короб (тап / скан) |
| `box.set_layout` | `{ boxId, rows: [{ rowId, kolPodb, kudaPodb, kolPerem, kudaPerem }] }` | сохранение раскладки строк короба |
| `box.full_to_ship` | `{ boxId, kudaPodb }` | флаг ПОЛН КОРОБ → весь короб уходит на один shipment |
| `box.change_address` | `{ boxId, novAdr }` | смена адреса физического короба |
| `box.inventory_correction` | `{ rowId, novKol, reason? }` | микро-инвент по конкретному баркоду в коробе |
| `ship.create` | `{ zayavkaId, taraType, number? }` → `{ shipBoxId }` | создание короба отгрузки |

| `request.set_active` | `{ zayavkaId | null }` | переключение активной заявки в сессии (фильтр) |

Каждый атом идёт в `SyncQueue`, имеет статус (`syncing → synced / error`) и отображается в UI индикатором сразу.

## 7. Тарификация подбора (free vs paid)

Решение принимается **на уровне физического короба-источника**, не строки.

Пусть для короба B:
- `units_to_ship(B)` = Σ `КОЛ ПОДБ` по всем строкам короба;
- `units_to_cell(B)` = Σ `КОЛ ПЕРЕМ`;
- `units_remaining(B)` = Σ `ОСТ`;
- `units_initial(B)` = `units_to_ship + units_to_cell + units_remaining` (= Σ `КОЛ` или `НОВ КОЛ`).

**Правило (free)**: короб **бесплатный**, если и только если
```
units_to_cell(B) == 0
AND units_remaining(B) == 0
AND units_to_ship(B) > 0
```
(физически: «всё, что было в коробе, ушло целиком на отгрузку, ничего не осталось и ничего не пошло в ячейки»). Эквивалентно: `ПОЛН КОРОБ = TRUE` для всех строк ИЛИ ручная раскладка дала тот же эффект.

**Правило (paid)**: иначе короб — **штучный подбор**.
```
charge(B) = units_to_ship(B) * RATE_PER_UNIT
```
где `RATE_PER_UNIT = 10 ₽/шт` (текущий тариф, см. `1_CONST/02 § 5.4`).

Важно:
- **Если из короба взяли ВЕСЬ один баркод**, а другой остался в источнике или ушёл в ячейку → короб **paid** (он был открыт). Это контр-интуитивно для подборщика, поэтому в UI badge короба меняется на `💰 Штучный` сразу при появлении не-нулевого `units_to_cell` или `units_remaining`.
- **Микро-инвент не влияет на тарификацию** — это отдельная операция (бесплатная, фиксируется в audit). Изменяет только `КОЛ`.
- Тарификация фиксируется при **завершении заявки** (логист закрывает подбор), а не построчно. До закрытия в UI — превью.

В БД храним для каждого «закрытия короба»:
```
PickEvent {
  boxId, zayavkaId, closedAt
  unitsToShip, unitsToCell, unitsRemaining
  isFree (computed)
  charge (computed)
}
```

## 8. Открытые вопросы

### 8.1. Где живёт `НУЖН` — РЕШЕНО

Источник = `ПОДБОРЫ.БД_ЭКСП.J` (`ЛОГ ЗАЯВКИ`). При старте приложения парсим J один раз, держим в памяти структуру `items: { barcode, qty }[]`. Вся логика «нужно/собрано/ещё» считается клиентом из этой структуры + локального состояния раскладки. Долгосрочно — `PickRequest` + `PickRequestItem` в БД WMS.

### 8.2. Источник коробов отгрузки активной заявки

Чтобы заполнять autocomplete `КУДА ПОДБ`, нужен список открытых коробов отгрузки заявки. Где?
- `🍬 КОРОБЫ` со статусом `СОБРАНО` / `В РЕЗЕРВЕ` и привязкой к заявке (поле F = `S0042` etc).
- `🚚 ОТГ` отдельно агрегирует.

→ Решение: фильтр по `🍬 КОРОБЫ.F = zayavkaId AND статус IN ('В РЕЗЕРВЕ','СОБРАНО','В УПАКОВКЕ')`. Если в Sheets живёт строго так — дублей не будет.

### 8.3. Идентификатор `КУДА ПЕРЕМ` (ячейка)

Ячейка хранения = `ЯЧ`-короб в `🍬 КОРОБЫ`, или отдельная таблица адресов? В CONST §3 `🍬 КОРОБЫ` тара `ЯЧ` фигурирует как тип короба. То есть **ячейка = тот же короб с открытой тарой**. Тогда autocomplete `КУДА ПЕРЕМ` = поиск по `🍬 КОРОБЫ` где `тара='ЯЧ' AND клиент=current`.

→ Подтвердить с пользователем.

### 8.4. Что делать с дублями в группе (bar5) — РЕШЕНО

Пользователь подтвердил: дубли — это **корректные две позиции в разных физических коробах** (тот же баркод в двух разных коробах). В UI это и должно быть так — две строки полотна, два независимых физических короба, в модалке открываются как разные.

### 8.5. Когда «закрывается» короб?

Кейс: подборщик открыл короб, разложил часть, закрыл модалку без `Сохранить`. Что в БД? Что в очереди sync?
→ Предлагаю: `Сохранить` обязателен; кнопка `Отмена` сбрасывает локальные правки; нет автосохранения по «закрыл вкладку».

### 8.6. Конкурентный доступ — РЕШЕНО (через разделение коробов отгрузки, не через локи)

**Контекст**: типичная заявка на 300 ед может собираться двумя–тремя подборщиками одновременно (так быстрее). Изначально предполагалось решать race-condition'ы через короткие локи на физический короб-источник.

**Решение**: разделение **по коробам отгрузки**, не по коробам-источникам.

1. Заявка `start` → в работу её берёт «ведущий» подборщик; статус → `В РАБОТЕ` с lock на ведущего (см. § 5.0).
2. Ведущий создаёт N коробов отгрузки батчем (§ 5.7) и печатает ленту этикеток.
3. Ведущий **физически отдаёт часть этикеток коллегам** (например, этикетки 11–20 — второму подборщику, 21–30 — третьему).
4. В UI каждый коллега заходит под своим аккаунтом, выбирает ту же заявку. Так как заявка уже `В РАБОТЕ`, обычный `picker` получает 409 Conflict с предложением «Подключиться как соподборщик» (атом `zayavka.join` — TBD).
5. После подключения соподборщик видит то же полотно, но в фильтре `КУДА ПОДБ` autocomplete показывает только коробы из его подмножества (определяется на бэке по `assignedShipBoxes`).
6. Конфликт «оба тапнули один короб-источник» резолвится короткими box-локами (timeout 10 мин с продлением, как было) — но это редкий случай (физически один короб у одной полки), а не основной паттерн.

**Что это даёт**:

- Параллельная работа без сложной координации — каждый отвечает за свой подмножество коробов отгрузки.
- Учёт собранного на уровне `PickEvent.actorId` — каждый короб отгрузки знает, кто его собирал → начисления per-сотрудник.
- Тарификация по-прежнему на уровне физического короба-источника (§ 7), независимо от того, сколько подборщиков касались заявки.

**Атом `zayavka.join`** (новый, дополняет § 6.1):

- Контракт: `{ zayavkaNumber, claimShipBoxes: number[] }` → `{ zayavka, joinToken }`.
- Доступен только если у заявки `status === 'В РАБОТЕ'` И запрашиваемые коробы отгрузки ещё не имеют `assignedTo`.
- Фиксирует `assignedTo[shipBoxId] = userEmail`. Подборщик-новичок становится со-владельцем заявки на время своих коробов.

### Sync engine — backend-driven таймеры

- `BATCH_TICK_MS = 120000` (2 мин) — flush очередей всех активных заявок одной пачкой.
- `REFRESH_TICK_MS = 600000` (10 мин) — re-read `🍬 КОРОБЫ` для forward sync.
- Микро-инвент / смена адреса / set_layout / ship.create — **через очередь** (2-минутный батч).
- `zayavka.start/finish/partial_close/close` — **немедленно** (записи в БД, обход очереди).

## 11. Стратегия внедрения: с Sheets-планшетов на web → ТСД

**Принцип**: один backend (атомы § 6.1), один UX-флоу (§ 5), но **адаптивный фронт** под устройство (§ 6.2–6.5). Не два разных приложения, а одно с режимами.

### 11.1. Текущее состояние склада (на 2026-04-28)

- Подборщики работают в **Sheets-таблицах «Планшет 1 / Планшет 2 / …»** (см. CONST/03 §5a). Заполняют 7 полей раскладки руками (`КОЛ ПОДБ / КУДА ПОДБ / КОЛ ПЕРЕМ / КУДА ПЕРЕМ / НОВ КОЛ / НОВ АДР / ПОЛН КОРОБ`). Чекбокс `ЗАЯВКА.H2` запускает `KorobyService_2.runFullUpdate` (4-шаговая транзакция).
- Физических ТСД на складе **пока нет**. Обещаны на будущее, но дата закупки и обучения сотрудников не определена.
- Этот web-проект разрабатывается в первую очередь как **замена Sheets-планшета**, не как ТСД-приложение.

### 11.2. Этапы перехода

**Этап P1. Web-планшет вместо Sheets-планшета** (текущая задача)

- Подборщик заходит на `/podbor/` с того же планшета (Android tablet, Chrome).
- Бизнес-флоу 1-в-1 как в Sheets-«Планшете», только UI современнее: тап по строке → BoxModal → numpad → Сохранить.
- Печать этикеток (§ 5.7a) — обязательный регламент с первого дня (это новое относительно Sheets-планшета).
- Sheets-«Планшет N» помечается deprecated, но физически остаётся доступным как fallback на случай инцидента.
- Параллельно сотрудники продолжают использовать `runFullUpdate` для финализации, пока web не начнёт писать в `🍬 КОРОБЫ` напрямую.

Критерий завершения P1: `KorobyService_2.runFullUpdate` больше не вызывается; все правки в `🍬 КОРОБЫ` + `🚚 ОТГ` + `НАЧ` + `ВР` идут с web-сервера.

**Этап P2. Гибрид: один-два ТСД параллельно с планшетами**

- Закупаем 1-2 ТСД (обычно Urovo / Honeywell, Android-based с физ. сканером и узким экраном ≤ 480 px).
- Тот же URL `/podbor/`, тот же логин, та же заявка. UI автоматически переключается в **ТСД-режим** на основании ширины экрана + presence сканера (см. § 6.5).
- Один-два подборщика-«пионера» работают на ТСД, остальные продолжают на планшетах. Учим 1-на-1, фиксируем UX-проблемы, итерируем UI.
- Параллельность через § 8.6: пионеры могут совместно работать с планшетчиками над одной заявкой — pool коробов отгрузки общий, разделение через `zayavka.join`.

Критерий перехода в P3: ≥80% сотрудников склада освоили ТСД, время сборки на ТСД ≤ времени на планшете.

**Этап P3. Только ТСД (планшеты в роли админ-инструмента)**

- Все подборщики на ТСД. Планшеты остаются для логиста / старшего смены / админа (десктоп 4:3 режим, § 6.2) — для контроля прогресса заявок и принудительного `unlock`/`override`.
- Sheets-«Планшеты N» удаляются физически (или переводятся в read-only архив).

### 11.3. Что общее, что разное между планшетом и ТСД

| Аспект | Планшет | ТСД |
|---|---|---|
| Бизнес-флоу | один и тот же (атомы § 6.1) | один и тот же |
| Стартовый экран | список карточек заявок (§ 5.0) | список карточек заявок, увеличенные кнопки |
| Открытие заявки | тап `[Начать]` | тап `[Начать]` ИЛИ скан QR с печатной заявки (если печатают) |
| Главный рабочий экран | полотно с группами BAR5 | вертикальный list-view: один короб-источник на экран, свайп между ними |
| Открытие короба-источника | тап по строке | скан QR на физическом коробе (если есть; иначе тап) |
| Выбор `КУДА ПОДБ` (короб отгрузки) | autocomplete + numpad короткий номер | **скан QR на этикетке короба отгрузки** → автоматическая подстановка; fallback — autocomplete |
| Выбор `КУДА ПЕРЕМ` (ячейка) | autocomplete | скан QR ячейки (тара `ЯЧ`); fallback — autocomplete |
| Numpad | экранный | физические клавиши ТСД работают параллельно |
| `+ Короба отгрузки` | модалка с numpad | то же + сразу печать ленты на ближайший термопринтер |

**Критичный инвариант**: НЕ должно быть бизнес-логики, которая работает на одном устройстве, но не работает на другом. Если фича невозможна на ТСД (например, из-за отсутствия экранного места) — она уходит в опциональный режим («Настройки» § 5.8) или в admin-only веб-интерфейс на десктопе.

### 11.4. Почему такой подход (vs два отдельных приложения)

**Аргументы за общий код**:

- Бизнес-инварианты (§ 4) и тарификация (§ 7) вычисляются на бэке — пишем один раз.
- Атомы (§ 6.1) идемпотентны и не зависят от устройства — один бэкенд обслуживает всех.
- Обновление UX-флоу (например, добавили шаг «комментарий причины» в `Завершить`) автоматически работает везде.
- Сотрудник, обученный на планшете, при выдаче ТСД видит тот же экран с теми же названиями кнопок — обучение на 80% переносится.

**Аргументы против общего кода (риски)**:

- ТСД-режим может выдавить решения, оптимальные для большого экрана (например, плотную таблицу). Митигация: в вертикальном list-view полотна на ТСД таблица не нужна — каждая строка = отдельная карточка.
- Может возникнуть соблазн сделать «универсальный» компонент, плохо работающий и там и там. Митигация: BoxModal остаётся идейно одна, но рендерит разные layout-шаблоны (`mode='tablet'` vs `mode='tsd'`) — `props` те же, разметка разная.

**Оценка**: преимущества общего кода многократно перекрывают риски в нашем размере склада (≤10 подборщиков, ≤30 заявок/день).

## 12. Финиш-pipeline и эксплуатационные правила (актуально на 2026-05-14)

### 12.1. Pipeline `zayavka.finish mode=full`

Шаги (исполняются параллельно в `runtime.js:582-637` через `Promise.all`):
1. **🍬 КОРОБЫ** (`UPSELLER_ID`) — массовая транзиция `В СБОРКЕ` → `СОБРАНО` через `finishZayavkaFull` под `sheetsMutex`.
2. **НАЧ** (`NACHISLENIYA_ID` — **отдельная таблица**, не путать с ПОДБОРЫ) — append строк начислений по `computed.nach.paidByBarcode`. **Идемпотентен**: перед append проверяет нет ли уже строк с этим `zayavkaId` (skip с `reason: 'already_written'`).
3. **readState** (локальный JSON) — синхронный read.

После Promise.all — последовательно:
4. **ПОДБОРЫ.БД** — `writeFinishSummary` пишет `U` (статус СОБРАНО), `V` (timestamp), `W:AJ` (24 поля сводки).
5. **archive** — state-файл переезжает в `_done/<id>-<ISO_ts>.json`. **Только при `writeFinishSummary.ok === true`** — иначе state остаётся в `zayavki/` и пользователь может продолжить.

Wall-time на проде: **5–10 сек** (параллельный pipeline). На dev с агрессивным batch-tick (≤5с) может растягиваться из-за Sheets API throttling (60 reads/min/user).

### 12.2. PERF-логи в stdout

Каждый из 4 шагов печатает `[finish:perf] <zayavkaId> <step>: <ms>` — для диагностики медленных финишей на проде.

### 12.3. Известные ограничения (open задачи)

- **Не-атомарность 3 шагов**: если шаг 4 (ПОДБОРЫ.БД `writeFinishSummary`) упал после шагов 1+2, КОРОБЫ уже переведены в СОБРАНО, НАЧ записан, а БД не помечена. Повторный finish: NACH skip'нет дубль (идемпотентен), `finishZayavkaFull` no-op (уже СОБРАНО), `writeFinishSummary` ретраит. Это semi-acceptable, но в долгосрочном плане нужен **pre-validation + true rollback** (см. `memory/project_podbor_archive_and_atomic_finish.md`).
- **Per-zayavka mutex deadlock** в `zayavka-store.js` лечится `LOCK_TIMEOUT_MS=30000` (Promise.race). Без таймаута одна зависшая операция блокировала всю заявку, рестарт сервера лечил. См. `memory/project_mutex_deadlock_zayavka_store.md`.
- **Sheets API rate limit**: read 60/min/user, write 60/min/user. На проде `PODBOR_BATCH_TICK_MS=10000` + `PODBOR_REFRESH_TICK_MS=60000` дают запас. Dev может ронять квоту при `BATCH_TICK_MS<10000` и >5 активных заявках — установить выше.

### 12.4. Нормализация баркодов (Озон-префикс)

`web/podbor/app.js:231-260` — `normBar(s)` оставляет только цифры. Озон в заявке указывает баркод как `3492817446`, на листе `🍬 КОРОБЫ` тот же товар — `OZN3492817446`. Все lookups (`requestedFor`, `pickedByBarcode`, `pickedByBar5`) сравнивают через `normBar` — иначе «не в заявке» / финиш-mismatch на каждом Озон-товаре.

### 12.5. Архив заявок (UI)

- Меню → 📜 **История заявок** → `/podbor/archive.html` (список с фильтрами клиент / даты)
- Клик по строке → `/podbor/archive-detail.html?zayavka=<id>` (полная карточка: events timeline, NACH, ship-коробы, source-коробы)
- Endpoint'ы: `GET /api/podbor/archive-list`, `GET /api/podbor/archive-detail`
- Источник: `${PODBOR_DATA_DIR}/_done/*.json` — append-only после успешного finish.
- Read-only: редактировать архив нельзя.
