> Этот документ — **стандарт**, не план реализации. Описывает целевое поведение приложения на ТСД и техническую модель интеграции. Реализация — в отдельном плане после согласования (см. § 16).

## 1. Целевое устройство — Urovo DT40

### 1.1. Аппаратные характеристики (из открытых источников Urovo)

| Параметр | Значение |
| --- | --- |
| Дисплей | 4″ TFT-LCD, **480 × 800 px** (WVGA), portrait по умолчанию |
| Яркость | до 550 nits, Gorilla Glass, capacitive touch (работает в перчатках) |
| Защита | IP67, дроп-тест 1.5 м |
| ОС | **Android 9 → 12 GMS** (зависит от ревизии — у пользователя надо уточнить) |
| Процессор | Qualcomm 1.8 GHz Octa-core 64-bit |
| Память | 2/16 GB или 3/32 GB |
| Батарея | 3.8 V / 4500 mAh, ~10 ч работы |
| Wi-Fi | 802.11 a/b/g/n/ac (5 ГГц поддерживается) |
| Сканер | Интегрированный 1D/2D scan engine (1D: EAN-13, EAN-8, UPC, Code-128, Code-39 и др.; 2D: QR, DataMatrix, PDF417, Aztec) |
| Физкнопки | 2× side scan keys (левая+правая, симметричные для левшей), top scan key, **full-keypad или numeric** (зависит от модели — DT40 / DT40S), функциональные клавиши |
| Прочее | NFC, BT 5.0, 4G LTE, GPS, фронтальная и задняя камеры |

> **Что уточнить у пользователя**:
> 1. Точная ревизия — DT40 или DT40S? Full-keypad (с QWERTY) или Numeric? Это влияет на наличие физических цифровых клавиш для ввода количества.
> 2. Версия Android — это определяет версию Chrome/WebView (≥ Android 7 = современный Chromium, всё ОК; Android 5–6 — потенциальные проблемы с CSS Grid и ES2020).
> 3. Конфигурация сканера — Wedge (по умолчанию) или Intent? См. § 4.

### 1.2. Браузер на DT40

DT40 идёт со штатным Chromium-WebView Android'а:
- Android 9 → Chrome 70+
- Android 12 → Chrome 95+

**Все современные web-API доступны**: ES2020, CSS Grid, fetch, Service Workers, IndexedDB, getUserMedia (для камеры). Ограничения — производительность и батарея (см. § 11).

## 1a. Тип заявки и режим сборки (общее для всех устройств, но критично для ТСД)

В `БД_ЭКСП` есть две дополнительные колонки, которые меняют процесс сборки:

| Колонка | Поле | Значения | Где хранится в JS |
| --- | --- | --- | --- |
| **P** | Тип заявки | `ОТГ` (отгрузка) / `ПЕР` (перемаркировка) | `z.type` |
| **Q** | Режим сборки | `СВОБ` / `КОР` / `КОР+` | `z.pickMode` |

### 1a.1. Тип заявки (`z.type`)

| Значение | Что это | Особенности процесса |
| --- | --- | --- |
| `ОТГ` (Отгрузка) | Стандартная клиентская отгрузка, итог едет на маркетплейс | Текущий основной флоу. После подбора короба идут на упаковку → ОТК → выезд. |
| `ПЕР` (Перемаркировка) | Товар изымается со склада, переклеивается ЧЗ-этикетка, возвращается обратно или уходит на ту же отгрузку | Источник заявок отличается, дальнейшая судьба коробов отличается. На MVP web-Подборы поддерживают подбор как часть процесса; финализация в UPSELLER может потребовать иной маршрут (пишется не в `🚚 ОТГ`, а в таблицу ПЕРЕМАРКИРОВКА). См. [`SYNC_BACKEND_PLAN.md`](SYNC_BACKEND_PLAN.md) § 2.2 «R-заявки» — этот флоу исторически делается отдельным GAS-проектом `ПЕРЕМ`. |

**На UI**: ОТГ — синий бейдж, ПЕР — фиолетовый. Виден всегда: на карточке заявки (старт), в шапке полотна, в шапке BoxModal. На ТСД — в шапке главного экрана и каждого подэкрана.

### 1a.2. Режим сборки (`z.pickMode`) — три сценария с разной логикой

Это **самое важное** поле для подборщика. Цвет в шапке должен бить в глаза, чтобы человек не перепутал режим.

#### СВОБ — Свободный подбор

- Можно брать поштучно из любого короба клиента (`КОЛ ПОДБ` < `КОЛ` разрешён).
- Можно создавать новые коробы отгрузки и микшировать в них товары из разных источников.
- Можно делать `КОЛ ПЕРЕМ` — внутреннее перемещение в ячейку.
- Микро-инвент разрешён.
- Это **базовый режим**, в котором web-Подборы работают сейчас.
- **Цвет**: зелёный.

#### КОР — Только полные короба

- **Запрещено** дербанить короба: можно брать ТОЛЬКО полностью (`КОЛ ПОДБ == КОЛ`).
- **Запрещено** микшировать: каждый короб клиента целиком становится коробом отгрузки. Никаких «новых коробов отгрузки», в которые подборщик складывает товар из разных источников.
- **Запрещено** `КОЛ ПЕРЕМ` (перемещение в ячейку) — это не подбор, а складская операция.
- **Разрешён** `Микро-инвент` (`НОВ КОЛ`) — но **до** подбора. Это пересчёт количества: подборщик открывает короб, видит что там не 50 а 48 → правит. Это не подбор, это уточнение остатка.
- **Цвет**: красный (самый строгий режим, ошибка дороже всего).

**Следствия для UI**:
- Кнопка `+ Создать коробы отгрузки` — **скрыта** или `disabled` с тултипом «В режиме КОР коробы создаются автоматически из коробов клиента».
- Слот `КОЛ ПОДБ` в карточке баркода — заблокирован с подсказкой «КОР: используйте `ПОЛН КОРОБ`».
- Слот `КОЛ ПЕРЕМ` — скрыт.
- Чекбокс/кнопка `ПОЛН КОРОБ` — единственный способ положить товар в подбор.
- При попытке нарушения — toast «В режиме КОР короба нельзя дербанить. Используйте полный короб или микро-инвент».

**На стороне backend `finalize`**: валидация `Σ kolPodb` строки = `qty` для каждой подбираемой строки. Любая строка с `kolPodb < qty` → отказ `400 KOR_MODE_VIOLATION`.

#### КОР+ — Приоритет коробам, остаток поштучно

**Точное правило (итеративное, проверяется на каждой попытке частичного подбора):**

Пусть для баркода `b` `stillNeeded(b) = requested(b) − Σ уже-собранного по b в других коробах`. Частичный подбор (`0 < kolPodb < qty`) для строки разрешён ТОЛЬКО когда выполнены ВСЕ три условия:

1. `qty > stillNeeded(b)` — иначе текущий короб помещается целиком, обязан идти `kolPodb = qty`.
2. Не существует **другого** не-дербаненного короба с тем же баркодом, у которого `qty ≤ stillNeeded(b)` — пока такие короба есть в наличии, дербанить запрещено (приоритет полным коробам).
3. Нет другого уже-раздербаненного короба для этого баркода (≤ 1 раздербаненный на баркод).

**Пример** (из обсуждения с пользователем): нужно 520 шт. Короба клиента: 10 шт по 50 + 1 короб 30 + 1 короб 80.
- На старте `stillNeeded = 520`. Все короба ≤ 520 → дербанить **запрещено**, обязаны брать полными.
- Берём 10 коробов по 50 → собрано 500, `stillNeeded = 20`.
- Короба остались: 30 и 80. Оба `> 20` → ни один не помещается целиком → **разрешено** дербанить ОДИН.
- Берём 20 из любого (например, из 30-короба). Заявка собрана. Второй короб (80) не трогаем.

**Что нельзя в этом сценарии:**
- Раздербанить два короба по 10. Запрещено правилом «3» (≤ 1 на баркод).
- Взять 30 из 80-короба, пока есть несобранный 50-короб. Запрещено правилом «2».
- Взять 40 из 50-короба, пока его qty ≤ stillNeeded. Запрещено правилом «1».

**Прочие ограничения КОР+:**
- `КОЛ ПЕРЕМ` запрещён (как и в КОР).
- Создание коробов отгрузки **разрешено** (для микса остатка).
- Микро-инвент разрешён.
- **Цвет**: оранжевый (промежуточный между СВОБ и КОР).

**Следствия для UI:**
- При попытке ввести `kolPodb` в зону частичного подбора, нарушающего правило → toast с конкретной причиной и подсказкой («Сначала возьмите целиком короб K10288 (50 ≤ нужно 70)»).
- Можно прокидывать визуальный hint в карточке баркода в BoxModal: «Этот короб помещается целиком (50 ≤ 520). Используйте 📦 ПОЛН КОРОБ» — но в MVP toast достаточно.
- Алгоритм-подсказка для подборщика (опц., поздняя фаза): кнопка `🪄 Авто-разложить КОР+` рассчитывает оптимальную раскладку «сначала максимум полных, потом остаток» и предлагает применить.

**Backend-валидация в `/api/podbor/finalize`** — порт того же правила (см. SYNC_BACKEND_PLAN_REVIEW § 7a):
- Для каждой строки с `0 < kolPodb < qty` проверить условия 1-3. Любое нарушение → `400 KORPLUS_MODE_VIOLATION { barcode, korob, kolPodb, qty, reason }`.
- Это защита от обхода UI (ТСД-сканер, вечно открытая вкладка с устаревшим pickMode).

### 1a.3. Где это видно в UI

| Место | Тип (ОТГ/ПЕР) | Режим (СВОБ/КОР/КОР+) |
| --- | --- | --- |
| Карточка заявки (стартовый экран) | компактный бейдж в `<div class="zc-tags">` | компактный бейдж + цветной край карточки слева |
| Шапка активной заявки (`zayavkaBar`) | бейдж 12-14px | **большой яркий бейдж**, КОР — белый текст на красном |
| Шапка BoxModal | small chip в углу | small chip с цветом |
| Главный экран ТСД | в шапке экрана | в шапке + visual-cue в footer (доступные действия меняются по режиму) |
| BoxModal на ТСД | в шапке карусели | **полоса цвета сверху** на полную ширину, чтоб подборщик не забыл |

### 1a.4. Default-значения

Если в `БД_ЭКСП.P/Q` пусто — нормализуем:
- `type` пусто → `ОТГ` (легаси, до введения колонки в мае 2026).
- `pickMode` пусто → `СВОБ` (свободный — наименее ограничивающий).

Это документировано в [`zayavki.js`](../../web/netlify/functions/_lib/podbor/zayavki.js) функциями `normalizeType()` / `normalizePickMode()`.

## 2. Принципы проектирования ТСД-интерфейса

1. **Скан > тап > свайп**. Каждое действие, которое можно сделать сканом, **должно** быть доступно сканом. Тап — fallback для подтверждений и numpad. Свайп — навигация по карточкам только.
2. **Один экран — одно действие**. На 480×800 не помещается «таблица + сайдбар + numpad одновременно». Каждый экран фокусирован на одной задаче.
3. **Тач-зоны ≥ 48×48 dp** (Material Design Touch Target). На 480px физических dp ≈ px, поэтому все интерактивные элементы — минимум 48px высотой, оптимально 56-64px.
4. **Шрифт ≥ 16px для тела**, 20-24px для заголовков, 28-36px для критических чисел (НУЖН/СОБР/ЕЩЁ). Не использовать `font-size: 11-13px` — нечитаемо на 4-дюймовом экране через стекло+перчатки.
5. **Контраст ≥ 4.5:1 (WCAG AA)**. На складе яркое освещение / тёмные углы — серый-на-сером не читается. Чёрный текст на белом фоне или белый на тёмно-синем для шапок.
6. **Никакого hover-state**. На ТСД нет мыши; всё что зависит от `:hover` — не работает или работает после тапа (sticky hover). Заменять на `:active` / `:focus-visible`.
7. **Минимум сети на действие**. ТСД на Wi-Fi → бывают разрывы в углах склада. Каждый атом (см. SYNC_BACKEND_PLAN § 4.1) — идемпотентный + retry-очередь на клиенте.
8. **Никаких модальных подтверждений на критичный путь**. «Вы уверены?» → подборщик нажимает «Да» не глядя. Лучше — undo-toast 5 секунд после действия.
9. **Звук + вибрация + цвет** на каждый скан (3 канала фидбэка):
   - Успех — короткий beep + 50ms виброотклик + зелёная вспышка.
   - Ошибка — двойной beep + 150ms вибро + красная вспышка + текст ошибки.
   На складе шумно, иногда виден только цвет, иногда только слышно — три канала избыточно но надёжно.
10. **Состояние всегда видно**. Шапка экрана = «Что я делаю сейчас» (`Подбор S1294-Видинеева`, `Короб K10234`, `Слот КУДА ПОДБ`). Подборщик за 2 секунды вернулся после прерывания и понял где он.

## 3. Detection — как приложение узнаёт, что оно на ТСД

Не стоит пытаться надёжно автоопределить ТСД (User-Agent сменяется, спецификации Wi-Fi не показывают `isTSD`). Лучшие подходы — в порядке предпочтения:

### 3.1. Ручной флаг «Я работаю на ТСД» (рекомендуется как primary)

В Settings (или один раз при первом входе) — toggle «Тип устройства»: `Десктоп / Планшет / Телефон / **ТСД**`. Сохраняется в `localStorage.podbor:device`. Приложение применяет соответствующий UI-режим.

Плюсы: явно, не ошибается, легко переключать (если планшет используется как «ТСД-замена»).
Минусы: нужен один setup-шаг при первом входе.

### 3.2. Авто-эвристика (как secondary, для default-выбора)

При первом входе — эвристика для подсказки дефолта:
- `navigator.userAgent` содержит `Urovo` или `DT40` → default = ТСД.
- `screen.width ≤ 480 && screen.height ≤ 854 && 'ontouchstart' in window` → подсказка «Похоже это ТСД, переключиться?».
- Иначе — default = планшет.

Юзер всегда может перебить вручную.

### 3.3. Query-параметр для теста

`http://192.168.0.165:3001/podbor/?device=tsd` — принудительно включает ТСД-режим в текущей вкладке. Удобно для разработки на десктопе через Chrome DevTools (Device Toolbar → 480×800).

### 3.4. CSS-стратегия

Не использовать чистые media queries для определения ТСД — на планшете с альбомной 1024×768 и портретной 480×854 ширина одинаковая ≈ 480 в портретной, но устройство разное. Применять `<html data-device="tsd">` через JS, и стили на `[data-device="tsd"] .xyz`.

## 4. Интеграция со сканером

### 4.1. Два режима сканера у Urovo

Urovo сканер на DT40 умеет работать в двух режимах (настраивается в системном `ScanWedge`/`Scanner Settings` приложении на ТСД):

| Режим | Как работает | Подходит web? |
| --- | --- | --- |
| **Wedge / TextBox** (default) | Сканер «печатает» данные в активный input как клавиатура (HID). По умолчанию добавляет `\n` (Enter) в конце. | ✅ ДА. Работает в любом браузере без интеграции. **Это наш основной режим.** |
| **Intent broadcast** | Сканер шлёт `android.intent.ACTION_DECODE_DATA` Broadcast Intent. | ❌ НЕТ напрямую. Web-страница в Chrome не получает Android Intent'ы. Требует нативной обёртки (см. § 14). |

**Решение**: всё web-приложение работает через **Wedge mode**. На ТСД при настройке проверяем, что Wedge включён (`PropertyID.WEDGE_KEYBOARD_ENABLE = 1`, по умолчанию это так).

### 4.2. Глобальный scanner-listener (JS)

Нужен один на всё приложение `window`-listener, который:
1. Накапливает символы в буфере.
2. По Enter (или таймауту 50 ms без новых символов) считает буфер «одним сканом».
3. Различает скан от ручного ввода клавиатуры по **скорости**: сканер выдаёт ≥100 chars/sec (10 ms между символами), человек ≤20 chars/sec.
4. Разрешает приоритет focused-input: если фокус в input типа «количество» (numpad), скан НЕ перехватывается глобально — пусть идёт в input как обычно (для Wedge так и нужно). Если фокус в `<body>` или вне input'а, глобальный listener забирает скан себе и роутит (см. § 5).

Псевдокод:
```js
let buf = '';
let lastT = 0;
window.addEventListener('keydown', (e) => {
  const now = Date.now();
  // Если прошло >100ms с последнего символа — новая «строка»
  if (now - lastT > 100) buf = '';
  lastT = now;

  if (e.key === 'Enter') {
    if (buf.length >= 6) handleScan(buf);  // ≥6 символов = баркод/код
    buf = '';
    if (shouldEatEnter()) e.preventDefault();
    return;
  }
  if (e.key.length === 1) buf += e.key;
});
```

Точные параметры (минимальная длина скана, таймаут, eat-enter-в-каких-input'ах) — настраиваются. Корректное поведение фокус/расфокус — самое тонкое место, тестируется отдельно.

### 4.3. Звук + вибрация после скана

Web API:
- Звук: `new Audio('/podbor/sounds/beep-ok.mp3').play()` (короткие предзагруженные mp3, ≤50 KB каждый).
- Вибрация: `navigator.vibrate(50)` (50 ms — успех, `[100,50,100]` — ошибка).

Оба API работают на Android Chrome ≥ 70 без разрешений. **Перед использованием** — `audio.preload = 'auto'`, иначе первый звук опаздывает на 200-500 ms.

### 4.4. Ограничения Wedge-режима, которые надо знать

1. **Сканер пишет ровно то, что закодировано в баркоде.** Если QR коробка отгрузки = `S1294-001`, и подборщик сосканировал её — приходит ровно строка `S1294-001\n`. Никаких префиксов от сканера нет. Префиксы — это ответственность того, кто печатает этикетку (см. § 5).
2. **Спецсимволы**: в barcode может быть `\n`, табы, кириллица. Wedge-режим Urovo всё это передаёт корректно, но кириллица иногда требует `Suffix language: Russian` в настройках. При первой настройке — проверяем на тестовом QR с кириллицей.
3. **Suffix Enter включён по умолчанию** (`Property: Suffix = 0x0A`). Не выключать — на нём держится наш detect.
4. **Beep самого сканера** — выключить (`Disable scanner beep`), потому что мы делаем свой beep с разделением OK/ERROR. Сканерский beep всегда «OK», даже если данные мусор.

## 5. Парсер скан-сообщений и роутинг

Чтобы один скан мог означать «короб-источник» / «короб отгрузки» / «ячейка» / «товар» — этикетки кодируют **разные префиксы**:

| Объект | Формат QR/баркода | Пример |
| --- | --- | --- |
| Короб-источник (физический короб клиента) | `K<NNNNN>` | `K10234` |
| Короб отгрузки (наш контейнер) | `S<NNNN>-<MMM>` (или `R<NNNN>-<MMM>` для R-заявок) | `S1294-001` |
| Ячейка склада | `Я:<код>` (двоеточие — обязательный разделитель) | `Я:A12-3-4` |
| Товар (баркод производителя) | EAN-13 / Code-128 (8-13 цифр) | `4607123456789` |
| Заявка (на печатном листе клиенту) | `Z:<zayavkaId>` | `Z:S1294-Видинеева` |

**Правило роутинга** в `handleScan(data)`:

```
1. Если data начинается с "K" + цифры → короб-источник → открыть BoxModal.
2. Если data начинается с "S<digits>-<digits>" или "R<...>-<...>" → короб отгрузки.
3. Если data начинается с "Я:" → ячейка.
4. Если data начинается с "Z:" → заявка → открыть полотно этой заявки.
5. Если data — голые цифры 8-13 символов → товар (баркод).
6. Иначе → unknown, beep-error, показать ошибку «Неизвестный код: ...».
```

**Контекст-зависимый роутинг** (что делать со сканом):
- **На стартовом экране** → `Z:` запускает заявку, всё остальное — игнор с пометкой «откройте заявку».
- **В полотне заявки** (без открытой модалки) → `K` открывает BoxModal этого короба, баркод товара — джамп к карточке этого баркода, остальное — игнор.
- **В BoxModal активной карточки** → `S` подставляется в `КУДА ПОДБ`, `Я:` — в `КУДА ПЕРЕМ`, баркод товара — джамп к другой карточке этого баркода в той же модалке, `K` — закрывает текущую модалку и открывает новую (другой короб-источник).

Подробная таблица «контекст × тип скана → действие» — отдельным приложением к стандарту (TSD_SCAN_MATRIX.md, делаем при реализации).

### 5.1. Кто отвечает за префиксы на этикетках

| Этикетка | Где печатается | Кто отвечает за префикс |
| --- | --- | --- |
| K-короб | Печатается приёмщиком при приёмке (PRIEMKA) | Внести в PRIEMKA-флоу: QR содержит `K<id>` (сейчас может быть просто `<id>` — проверить и привести к стандарту) |
| S-короб (отгрузка) | Печатается web-Подборами при `ship.create` (термопринтер 58×40) | Уже работает — генерируем `S1294-001` (см. `shipBoxQRCache`). Префикс `S` уже там. ✅ |
| Я-ячейка | Печатается отдельным проектом ТСД-адресации | Согласовать со складом: вся ячеечная маркировка должна быть `Я:<код>` |
| Z-заявка | Печатается с распечаткой ТЗ (если печатают) | Сейчас не печатаем — добавить, если решим использовать |

> **Действие до запуска**: согласовать со складом, что **все** новые этикетки печатаются с префиксами по этому стандарту. Существующие — массово **не** перепечатывать (стоимость), но web должен поддерживать **обратную совместимость**: голый баркод 8-13 цифр считается товаром, голый `<число>-<число>` — устаревшая S-маркировка → принимается с warning «обновите этикетку».

## 6. UI-стандарт для ТСД-режима

### 6.1. Глобальное

| Параметр | Значение |
| --- | --- |
| Viewport | `<meta name="viewport" content="width=480, initial-scale=1, maximum-scale=1, user-scalable=no">` (фиксированная ширина — стабильный layout, никакого pinch-zoom) |
| Корневой шрифт | `font-size: 16px` на `<html>` (база для rem) |
| Шрифт тела | `16-18px` (1-1.125 rem) |

| Критические числа (НУЖН/СОБР/ЕЩЁ, qty в карточке) | `28-36px`, моноширинный (`Roboto Mono`/`JetBrains Mono`) |
| Цвет текста основной | `#111` на `#fff` |
| Цвет акцента | `#0a5` (зелёный — успех/подтверждение) |
| Цвет ошибки | `#c33` (красный — error, миссматч) |
| Цвет предупреждения | `#e90` (оранжевый — partial / нужно внимание) |
| Тач-зона минимум | `48×48 px` (целевая 56×64) |
| Высота кнопки primary | `64px` |
| Радиус скругления | `12px` (крупнее чем на десктопе — пальцем легче целиться) |

### 6.2. CSS-брейкпоинт ТСД

Сейчас в [`style.css`](local-dev/public/podbor/style.css) есть `@media (max-width: 479px)`. Для DT40 **480 × 800** — это ровно 480 = на границе. Стандарт:

```css
/* ТСД-режим: data-attribute, не media query */
[data-device="tsd"] { ... }

/* Дополнительный safety — узкий экран */
@media (max-width: 480px) {
  /* Резервные стили для случая когда data-device не выставлен */
}
```

Используем `[data-device="tsd"]` как **основной** селектор, media query — fallback. Это позволяет принудительно включить ТСД-режим на планшете для тестирования (через Settings или query param).

### 6.3. Шапка приложения (header) на ТСД

Компактная: 56px высоты, фиксированная top-sticky.
Содержит:
- Слева: кнопка `←` (назад) или меню `≡`.
- Центр: текущий контекст (`Подбор S1294-Видинеева` / `Короб K10234`).
- Справа: индикатор сети (зелёная/жёлтая/красная точка) + батарея ТСД (если PWA → `navigator.getBattery()`).

**Никакого** аватара/имени пользователя в шапке — занимает место. Имя — в меню `≡`.

### 6.4. Footer / нижняя панель

На ТСД нижняя панель = **главные действия текущего экрана** (sticky bottom, 80px высоты, две-три большие кнопки):
- В полотне заявки: `Завершить заявку` / `+ Создать коробы` / `Печать этикеток`.
- В BoxModal: `Сохранить` / `Отмена` / `Микро-инвент`.

Кнопки **большие**, цвета чёткие, иконка + подпись (16px).

## 8. BoxModal на ТСД

На планшете BoxModal — full-screen, с таблицей баркодов в коробе и numpad сбоку.

На ТСД таблица не помещается. Замена — **вертикальная карусель карточек**: одна карточка на экран, свайп влево/вправо для следующего/предыдущего баркода. Внутри карточки:

```
┌───────────────────────────────────┐
│ ← Короб K10234 (3/8)            ✕ │ ← заголовок: «короб + позиция в карусели»
├───────────────────────────────────┤
│ Баркод: 4607123 |456789|          │ ← последние 5 крупнее
│ SKU: PLATE-WHITE-30                │
│ В коробе: 12 шт   Ячейка: А12-3   │
├───────────────────────────────────┤
│ КОЛ ПОДБ:                         │
│ [ - ] [    5    ] [ + ]           │ ← numpad большой, 64px кнопки
│ КУДА: [📷 Скан короба отгрузки]   │
│       Сейчас: S1294-001           │
├───────────────────────────────────┤
│ КОЛ ПЕРЕМ:                        │
│ [ - ] [    0    ] [ + ]           │
│ КУДА: [📷 Скан ячейки]            │
├───────────────────────────────────┤
│ [Микро-инвент] [Сохранить] [Далее]│ ← 3 кнопки в footer
└───────────────────────────────────┘
```

**Свайп влево** = «Сохранить + Далее» (следующая карточка). Подборщик расскладывает короб баркод-за-баркодом, не выходя из модалки.

**Скан внутри модалки**:
- Скан баркода товара → джамп к этой карточке (если она есть в коробе) или beep-error.
- Скан `S1294-XXX` → подставляется в `КУДА ПОДБ` активной карточки.
- Скан `Я:XXX` → подставляется в `КУДА ПЕРЕМ`.
- Скан другого `K` короба → `Сохранить + Закрыть текущий + Открыть новый` (одно действие).

Чекбокс `ПОЛН КОРОБ` — внизу модалки (как сейчас), но выше footer'а.

## 9. Поток подбора на ТСД (полный сценарий)

Сценарий: подборщик собирает заявку `S1294-Видинеева`, нужно 247 шт по 5 SKU из 8 коробов клиента.

1. Логин (Google OAuth, как обычно). На ТСД — выбор аккаунта из списка ранее использованных (`localStorage`-cache).
2. Стартовый экран — вертикальный список карточек заявок (как на планшете, см. CONTEXT.md § 5.0). Можно отсортировать только «мои в работе».
3. Тап на карточку или **скан `Z:S1294-Видинеева`** с распечатки ТЗ → открывается **главный экран ТСД** (см. § 7).
4. Подборщик идёт к стеллажу, находит первый короб клиента, **сканит K-этикетку** → BoxModal этого короба (см. § 8).
5. Внутри:
   - Сканит баркод товара → карточка нужного баркода в фокусе.
   - Жмёт `+ +5` (или вводит 5 в numpad).
   - Сканит S-этикетку первой пустой коробки отгрузки на тележке → подставляется `КУДА ПОДБ = S1294-001`.
   - Свайпает влево → следующая карточка короба.
6. Когда все строки короба разложены — `Сохранить` (физическая кнопка ТСД может быть замаплена на это).
7. Идёт к следующему K-коробу, сканит его → новый BoxModal.
8. Если в процессе понял что в коробе на самом деле не 12 шт а 10 (микро-инвент) → кнопка `Микро-инвент` → numpad → причина (preset-ы).
9. Когда всё собрано — на главном экране кнопка `✅ Завершить заявку` → подтверждение → `POST /api/podbor/finalize` (см. SYNC_BACKEND_PLAN). Ответ показывает чек-лист `✅ КОРОБЫ / ✅ ОТГ / ✅ НАЧ / ✅ БД`.
10. Если упало — toast «Не удалось завершить, ошибка X». Заявка остаётся в работе, ничего не теряется.

## 10. Создание и печать коробов отгрузки на ТСД

**Сценарий**: подборщик у тележки, нужно создать 5 пустых коробов отгрузки.

1. Главный экран ТСД → `+ Создать коробы отгрузки`.
2. Модалка: numpad «Сколько?» (1-10 типичные) + select типа тары (`К_0.5 / К_1.0 / ПАЛ`).
3. Жмёт `Создать` → атом `ship.create` → backend генерит номера + QR.
4. Модалка показывает 5 номеров `S1294-001 … S1294-005` с QR-плитками.
5. Кнопка `🖨 Печать` → отправка на термопринтер. **Здесь важно**: ТСД не подключён к термопринтеру USB-кабелем. Варианты:
   - **A**: термопринтер по Bluetooth, спарен с ТСД. Web использует `Web Bluetooth API` (Chrome для Android поддерживает с 2018). Сложно, требует разрешений + UUID принтера.
   - **B**: термопринтер по Wi-Fi (network-printer). ТСД делает HTTP POST на IP принтера с ZPL/EPL-кодом. Проще, но принтер должен быть в той же сети.
   - **C** (проще всего на старте): кнопка `🖨 Печать` → backend шлёт задание на печать на сервер, который печатает на физически подключённом термопринтере (на десктопе оператора склада). ТСД не печатает, но запускает печать.

**Решение для MVP**: вариант C. Дальше — в зависимости от того, как далеко подборщик от принтера на практике.

## 11. Производительность

### 11.1. WebView ограничения на DT40

- ОЗУ 2-3 GB, выделено WebView ~256-512 MB. **Не держать в памяти 1000+ DOM-узлов.** Виртуализация списков обязательна для коробов клиента (50+ K-коробов на одного клиента — реально).
- CPU 1.8 GHz (8 ядер), но энергосберегающие режимы режут одно-поточные задачи. Тяжёлые синхронные циклы > 200 ms = заметный лаг.
- Нет аппаратного декодирования некоторых видео-форматов — но нам не нужно видео.

### 11.2. Что пересмотреть в текущем коде

| Что | Сейчас | Для ТСД |
| --- | --- | --- |
| Рендеринг полотна заявки | Один `innerHTML` всех групп баркодов | Виртуализация (рендерим только ±20 групп от viewport) |
| Кэш QR-этикеток (`shipBoxQRCache`) | In-memory `Map<number, dataUrl>` | Уносить в IndexedDB, не держать 100 base64 PNG в JS-heap |
| Шрифты | Загрузка `Inter` 4 веса (≈ 160 KB) | Один `Inter Regular` + `Bold`, или системный `-apple-system, Roboto` (0 KB) |
| CSS-анимации | Полный набор transitions | На ТСД: `[data-device="tsd"] * { transition: none !important; }` для главных элементов; анимация только для модалок (open/close) |

### 11.3. Wi-Fi разрывы

Склад — большое металлическое помещение, Wi-Fi может пропадать в углах. Стратегия:

1. **Каждый атом** (`box.set_layout`, `ship.create`, `inventory_correction`) идёт через **очередь с retry** на клиенте. Если 5xx или таймаут — кладём в `localStorage`, retry экспон. бэкоффом, индикатор `syncing` в UI.
2. **Финализация заявки** (`/api/podbor/finalize`) — НЕ retry. Это критическое одноразовое действие; повтор — только через явный клик подборщика (с idempotency-key, см. SYNC_BACKEND_PLAN § 4.3).
3. **Чтение** (`/api/load`, `/api/zayavki-list`) — fail-fast, показать «Нет связи. Повторить?».

### 11.4. PWA?

Имеет смысл оформить web-Подборы как PWA:
- Manifest (`manifest.json`) — иконка на главном экране ТСД, fullscreen без браузерного UI.
- Service Worker — оффлайн-кэш статики (HTML/JS/CSS), graceful degradation при пропадании сети.

**Это не делаем сейчас**, но архитектурно учесть: статика отделена от API, нет inline-state в HTML — всё это уже выполнено.

## 12. Сетевые сбои и offline-поведение

| Ситуация | Поведение |
| --- | --- |
| Wi-Fi пропал во время `box.set_layout` | Атом в retry-очереди, индикатор `syncing` (жёлтый), UI остаётся работоспособным (раскладка в localStorage). |
| Wi-Fi пропал перед `finalize` | Кнопка «Завершить» = `disabled`, тост «Дождитесь подключения». При восстановлении — кнопка снова активна. |
| `finalize` упал на полпути | Backend-rollback (см. SYNC_BACKEND_PLAN § 3.8). Клиент получает `500 ROLLBACK_OK` или `500 DESYNC` — показывает соответствующий тост. |
| ТСД полностью offline > 5 минут | Принудительный logout с тостом «Соединение потеряно — войдите снова». Накопленные атомы остаются в localStorage; при логине — спрашиваем «Восстановить незавершённый подбор?». |

## 13. Тестирование

### 13.1. Что тестируется на эмуляторе/десктопе

- UI-режим через `?device=tsd` или Settings → ТСД.
- Chrome DevTools → Device Toolbar → custom 480×800.
- Скан эмулируется быстрой подачей `keypress`-событий через DevTools-снippet или клавиатуру:
  ```js
  // Snippet: эмулировать скан "S1294-001\n"
  function emulateScan(s) {
    [...s, '\n'].forEach((ch, i) => setTimeout(() => {
      const ev = new KeyboardEvent('keydown', { key: ch === '\n' ? 'Enter' : ch });
      document.activeElement.dispatchEvent(ev);
    }, i * 5));
  }
  ```

### 13.2. Что тестируется только на реальном DT40

- Реальный сканер в Wedge-режиме (тайминги, suffix Enter, поведение фокуса).
- Кириллица в QR-коде (если есть).
- Beep + vibrate синхронность.
- Жесты свайпа на capacitive touch с грязью / в перчатках.
- Wi-Fi roaming между точками доступа на складе.
- Батарея за смену (8 ч непрерывной работы).
- Падение на бетон с 1 м (это IP67 заявленные, но проверить именно эту партию).

### 13.3. Тест-чек-лист (минимум для раскатки)

## 14. Когда упрёмся в потолок web — Native bridge

Если после внедрения окажется, что:
- Wedge-режим теряет фокус слишком часто (нужен Intent broadcast),
- Web Bluetooth не работает с конкретным термопринтером,
- WebView лагает на больших полотнах (>200 групп),
- Нужны нативные функции (NFC-сканирование сотрудника, GPS-привязка ),

— тогда **тонкая нативная Android-обёртка**, которая открывает наш сайт в WebView и пробрасывает Urovo SDK через `addJavascriptInterface`. Архитектурно: 95% кода остаются web (один codebase для планшета и ТСД), 5% — Kotlin/Java-обёртка с broadcast-receiver `ACTION_DECODE_DATA` → `webView.evaluateJavascript('window.__nativeScan(...)')`.

**Это решение не для MVP.** Запускаем чистый web с Wedge — этого хватит для 80% сценариев. Нативная обёртка — отдельный этап после P2 в [`local-dev/CONTEXT.md`](local-dev/CONTEXT.md) § 11.2.

## 16. Открытые вопросы (перед стартом реализации)

1. Точная ревизия DT40 (full-keypad / numeric / DT40S)?
2. Версия Android на конкретном устройстве?
3. Сканер уже настроен в Wedge-режиме? Suffix = `\n`? Кириллица проверена?
4. Что физически делает левая/правая/верхняя scan-кнопка сейчас? Можно ли замапить какую-то на «Сохранить раскладку» через `KeyMapManager`?
5. Префиксы на этикетках: текущие K-этикетки имеют префикс `K` или просто число? Аудит существующих этикеток до запуска.
6. Термопринтер на складе — какая модель, как подключён (USB к десктопу / BT / Ethernet)? От этого выбор варианта печати в § 10.
7. Сколько коробов клиента типично в самой большой заявке (для оценки виртуализации)?
8. Сколько подборщиков одновременно на одном складе (для оценки конкуренции при lock)?
9. Кириллица в SKU/баркодах товаров встречается? Если да — Wedge должен корректно её передавать.
10. PWA на главный экран — нужно или Chrome-вкладка достаточна? (PWA = меньше отвлекающего UI, но требует HTTPS на VPS).
