# План синхронизации Podbor → UPSELLER (краткая версия для ревью)

> Полная версия со всеми деталями колонок/валидаций/rollback-сценариев — [`SYNC_BACKEND_PLAN.md`](SYNC_BACKEND_PLAN.md). Этот документ — выжимка для принятия решения «делаем / меняем / откладываем».

## 1. Что меняем

**Сейчас**: подборщик работает в Sheets-Планшете → нажимает чекбокс на листе ЗАЯВКА → GAS-скрипт `KorobyService_2.runFullUpdate` пишет в UPSELLER (КОРОБЫ + ОТГ) и ПОДБОРЫ (БД + ВР) и НАЧИСЛЕНИЯ (НАЧ).

**Будет**: подборщик работает в web-планшете (`/podbor/`) → нажимает «Завершить заявку» → backend сам пишет ровно в те же 4 листа в той же последовательности. Sheets-Планшет (лист ПОДБОР) уходит из контура полностью.

## 2. Текущая проблема

Web-планшет уже умеет читать UPSELLER, показывать коробы клиента, принимать раскладку (kolPodb, kudaPodb, kolPerem, kudaPerem) и создавать коробы отгрузки — но **всё это копится только в памяти процесса** (`boxLayoutStore`, `shipBoxStore`, `inventoryOverrides`). При рестарте — теряется. В UPSELLER ничего не попадает.

Финализации заявки нет ни в каком виде. Пока подборщик не может довести заявку до конца через web — ему всё равно нужно идти в Sheets-Планшет.

## 3. Решение в одной фразе

Один новый эндпоинт `POST /api/podbor/finalize` — порт GAS-функции `runFullUpdate` 1-в-1: 4 шага транзакции (КОРОБЫ → ОТГ → НАЧ → БД) со snapshot+rollback, плюс отчёт в `ВР`.

## 4. Архитектура (ключевые решения)

| Решение | Почему |
| --- | --- |
| Атомы (`box.set_layout`, `ship.create`, `inventory_correction`) **продолжают писать только в memory**, не в Sheets | Так же как GAS — все правки накапливаются, потом одной транзакцией. Меньше API-вызовов, проще rollback. |
| Backend **сам считает 4 структуры записи** (updates/newRows для КОРОБЫ, otg, nach, bd) | Нет промежуточного листа `РАСЧЁТ`. Логика повторяется на JS из GAS-формул. |
| Lock на заявку через `@netlify/blobs` (TTL 120 сек) | Защита от параллельных сессий двух подборщиков. |
| Idempotency-Key (UUID от клиента) с кэшем 24 часа | Защита от двойного нажатия / повторного запроса. |
| Writer **только через OAuth refresh-token** (`google.js`), не через сервис-аккаунт | SA сейчас readonly. Refresh-token от `psgl2007@gmail.com` имеет полные права. |
| **R-заявки (перемаркировка) на MVP не поддерживаем** | Они идут через отдельный GAS-проект `ПЕРЕМ`, у нас focus только на S-заявки. |
| Сортировка `🍬 КОРОБЫ` после finalize — **пропускаем на MVP** | Не критично, можно добавить позже. |
| Запись в НАЧ — **одна агрегатная строка** (Variant B) | Полный детальный набор (как у GAS из формул `РАСЧЁТ.EI:ET`) — позже. |

## 5. Этапы (порядок реализации)

| # | Этап | Оценка | Что проверяем |
| --- | --- | --- | --- |
| 1 | **Read-only обвязка**: `upseller-writer.js` skeleton + `calc-updates.js` + `POST /api/podbor/preview` (возвращает «что бы записал», но не пишет) | 2-4 ч | На тестовой заявке preview возвращает корректные updates/newRows |
| 2 | **Один сценарий end-to-end**: «частичное изъятие на отгрузку» — реальная запись в `🍬 КОРОБЫ` | 4-6 ч | В тестовой строке `I` стало `qty - kolPodb`, новая строка появилась |
| 3 | **Остальные сценарии в `🍬 КОРОБЫ`**: `kolPerem` (PEREM_V_YACH/SOZD_YACH), микро-инвент (NOV_QTY), `KOROBA_OTGRUZ` | 3-4 ч | Все 6 сценариев из GAS работают |
| 4 | **ОТГ + БД + ВР**: writeOtgStep, writeBdStep, writeVrStep | 2-3 ч | Заявка закрывается, отчёт пишется в `ВР` |
| 5 | **НАЧИСЛЕНИЯ + Idempotency + Audit** | 2-3 ч | Повторный finalize не дублирует записи |
| 6 | **Rollback + конфликты + 🆘** | 3-4 ч | При фейле любого шага состояние возвращается к началу |
| 7 | **Отказ от mem-store, переход на blob** (опц.) | 3 ч | Раскладка переживает рестарт сервера |

**Итого**: 19-27 часов на полную реализацию. После этапа 4 — **уже можно использовать** в продакшене на одного-двух подборщиков (без идемпотентности и rollback это рискованно, но физически работает).

## 7. Риски (топ-3)

1. **OAuth scope недостаточный** → не сможем писать. Митигация: проверить на этапе 1, при необходимости перевыпустить refresh-token со scope `spreadsheets`.
2. **Rate-limit Sheets** при больших заявках → 429. Митигация: ретраи с экспон. бэкоффом, lock-per-заявка уже решает race.
3. **Параллельная работа в Sheets-Планшете и web-Планшете на одной заявке** → drift. Митигация: lock + чёткое сообщение «Заявка занята другим подборщиком» + флаг в `БД_ЭКСП.O = В РАБОТЕ` при первом `box.set_layout`.

## 7a. Учёт режима сборки (`БД_ЭКСП.Q`)

Заявка несёт два новых поля:
- `z.type` (`БД_ЭКСП.P`) — `ОТГ` / `ПЕР`. На MVP подбираем одинаково; различие — в финализации (ПЕР=R-заявки идут не в `🚚 ОТГ`, а в таблицу ПЕРЕМАРКИРОВКА).
- `z.pickMode` (`БД_ЭКСП.Q`) — `СВОБ` / `КОР` / `КОР+`. Меняет логику валидации в `/api/podbor/finalize`.

**Что добавить в этап 1 (`calc-updates.js` / preview)**:

| Режим | Дополнительная валидация перед записью |
| --- | --- |
| `СВОБ` | Текущая логика без изменений. |
| `КОР` | Для каждой подбираемой строки: `kolPodb ∈ {0, qty}`. Любое `0 < kolPodb < qty` → `400 KOR_MODE_VIOLATION { korob, barcode, kolPodb, qty }`. `kolPerem == 0` для всех строк (внутренние перемещения запрещены). UI шлёт `kudaPodb = '__AUTO_KOR__'` (sentinel) — backend на финализации сам генерирует один `S<NNNN>-<MMM>` на каждый исходный K-короб (один-к-одному, без `KOROBA_OTGRUZ`-блока для нового пустого контейнера). См. § 7b. |
| `КОР+` | Итеративная проверка для каждой строки с `0 < kolPodb < qty` (частичный подбор). Пусть `stillNeeded(b) = requested(b) - Σ kolPodb(b) других строк`. Запрещено частично подбирать строку если: **(а)** `qty ≤ stillNeeded(b)` (короб помещается целиком — обязан идти `kolPodb=qty`), **или (б)** существует другой не-дербаненный короб с тем же `barcode` где `qty ≤ stillNeeded(b)` (приоритет полным коробам), **или (в)** уже есть другой раздербаненный короб для этого баркода (≤ 1 partial на баркод). Любое нарушение → `400 KORPLUS_MODE_VIOLATION { barcode, korob, kolPodb, qty, reason }`. `kolPerem == 0`. |

**Default**: пустые `P/Q` нормализуются в `ОТГ` / `СВОБ` (легаси-строки).

См. подробности по UI и сценариям подборщика — [`TSD_STANDARD.md`](TSD_STANDARD.md) § 1a.

## 7b. Sentinel `__AUTO_KOR__` и auto-create S-номеров в КОР

В режиме КОР подборщик не выбирает целевой короб отгрузки — он один на весь короб клиента. UI хранит в `draft.kudaPodb = '__AUTO_KOR__'` для всех строк короба. На финализации backend:

1. Группирует все строки с `kudaPodb === '__AUTO_KOR__'` по `boxId` (исходному K-коробу).
2. Для каждой такой группы выделяет следующий свободный номер `S<zayavkaPrefix>-<NNN>` (или `R<...>-<NNN>` для R-заявок), используя `state.shipBoxStore[zayavkaId].nextSeq` или, если store пуст, начиная с `001`.
3. Подменяет `kudaPodb` на сгенерированный `S<...>-<NNN>` перед расчётом updates.
4. Для каждого такого S-короба создаёт строку `KOROBA_OTGRUZ` (как в § 5.4) с `taraType` = тара исходного K-короба (поле `B` строки в `🍬 КОРОБЫ`).

Это даёт «один-в-один»: каждый K-короб клиента → один S-короб отгрузки той же тары. Подборщик ничего не создаёт и не выбирает.

**Идемпотентность**: если finalize ретраится с тем же `Idempotency-Key`, генерируем те же S-номера — backend хранит mapping `K<id> → S<id>` в blob-аудите.
