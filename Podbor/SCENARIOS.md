# Сценарии Подбора — что куда пишем

Источник правды бизнес-логики:
- `1_CONST/02_BUSINESS_PROCESSES.md` § 5 (тарификация подбора), § 5b (тара-коэф, КЛ/ФФ)
- `Podbor/CONTEXT.md` § 6 (атомы), § 7 (free/paid)

Здесь — компактная карта **«сценарий → атом → мутации в листах»**. Без воды.

## Соглашения (структура `🍬 КОРОБЫ`)

Колонки (range A:U + V/X/Y):
- `A=БАР_5` — формула, **не трогаем**
- `B=ДАТА СОЗ` — DD.MM.YY (наша запись при создании/трансформации)
- `C=ТАРА` — коэффициент: `К_1.0`/`К_0.5`/...
- `D=КОРОБ` — номер
- `E=СТАТУС` — `ХРАНЕНИЕ`/`ГОТОВО`/`В СБОРКЕ`/`СОБРАНО`/`ОТГРУЖЕНО`
- `F=ЗАЯВКА` — № подбора (`S0042-Клиент`)
- `G=ТИП` — `УТ ГОТОВ`/`БРАК`
- `H=SKU`, `I=КОЛ`, `J=АДР`
- `M=СКЛАД НАЗН` — соединение `<warehouse> ▹ <finalWarehouse>` если оба, иначе один
- `N=ДАТА ОТГ` — DD.MM.YY (источник: БД_ЭКСП.E)
- `R=КОММЕНТАРИЙ` — multiline: габариты, КЛ/ФФ, прочие пометки
- `S=МП`, `T=КЛИЕНТ`, `U=БАРКОД`
- `X=№ ОТГ` — **равен номеру подбора** для UPSELLER-флоу (подбор = часть отгрузки)
- `Y=ДАТА ОТГ` — (резерв)

## Источники данных при записи короба отгрузки

При создании/трансформации короба заполняем эти поля **из БД_ЭКСП заявки**:

| КОРОБЫ.поле | Источник |
|---|---|
| `F` (ЗАЯВКА) | `БД_ЭКСП.C` (№ подбора) — например `S0042-Клиент` |
| `X` (№ ОТГ) | то же значение, что и F |
| `M` (СКЛАД НАЗН) | `БД_ЭКСП.G` (склад) `▹` `БД_ЭКСП.H` (конечный склад). Если H пусто — только G. |
| `N` (ДАТА ОТГ) | `БД_ЭКСП.E` (дата отгрузки), формат DD.MM.YY |
| `S` (МП) | `БД_ЭКСП.F` (МП заявки) или копия из исходной строки |
| `T` (КЛИЕНТ) | имя клиента из заявки |

## Содержимое R (КОММЕНТАРИЙ) — multiline

Каждый короб отгрузки должен иметь в R:
1. **Габариты** короба-микса в см: `60×40×40` (стандарт). Из габаритов выводим коэф тары для C.
2. **Признак владельца тары**: `КЛ` (клиента) или `ФФ` (фулфилмента) — обязательно для коробов отгрузки и коробов полного изъятия.
3. (Доп.) другие служебные пометки, разделяем `\n`.

Пример R:
```
60×40×40
КЛ
```

## Соглашения

- `<src>` = строка короба-источника, `<dst>` = строка короба отгрузки или ячейки.
- `qty(x)` = текущее `КОЛ` в строке x.

## A. Создание короба отгрузки `ship.create` (микс-короб)

**Trigger**: «+ Короб отгрузки» в UI.
**Атом**: `ship.create { count, taraType?, dimensions?, owner?: 'КЛ'|'ФФ' }`.
**UX до создания**: спросить **габариты Ш×В×Г см** + **владельца тары (КЛ/ФФ)**. Из габаритов вычислить коэф тары: `К_volume/(60×40×40)` округлённое до 0.5 или 1.0 (`К_1.0` стандарт, `К_0.5` пол-объёма).
**Сразу**: ничего на лист не пишем. Короба живут как **виртуальные** в очереди (`queue.shipBoxes`), QR генерится на бэке.
**На лист попадают** только при первом материале (см. D). Если до финиша короб остался пустым → удаляется (`ship.delete`), на лист не попадает.

## B. Полное изъятие короба `ПОЛН КОРОБ` — **трансформация исходной строки**

**Trigger**: модалка короба, кнопка «Изъять полный короб». **Не создаём новый короб отгрузки** — короб-источник сам становится коробом отгрузки.
**UX до подтверждения**: спросить **владельца тары (КЛ/ФФ)** — записывается в R источника. Габариты не нужны (тара от источника не меняется, T (тара) остаётся).
**Атом**: `box.full_to_ship { boxId, owner: 'КЛ'|'ФФ' }`.
**Sync flush** (трансформируем `<src>` строку каждого баркода в коробе):

| Колонка | Действие |
|---|---|
| `E` (СТАТУС) | `:= В СБОРКЕ` |
| `F` (ЗАЯВКА) | `:= zayavkaId` (`S0042-Клиент`) |
| `M` (СКЛАД НАЗН) | `:= <warehouse> ▹ <finalWarehouse>` |
| `N` (ДАТА ОТГ) | `:= dateOtgr` |
| `R` (КОММЕНТ) | append `\nКЛ` или `\nФФ` |
| `X` (№ ОТГ) | `:= zayavkaId` |

`КОЛ` (`I`) **не трогаем** — он остался прежним. `D` (КОРОБ) — тот же физический номер.

**Тарификация**: free (charge=0) — короб ушёл целиком на отгрузку.
**ВР / НАЧИСЛЕНИЯ**: на финише заявки.

## C. Перекладывание в ячейку `move to cell`

**Trigger**: модалка, для конкретного баркода `kolPerem=N, kudaPerem=<cell>`.
**Атом**: `box.set_layout` с `kolPodb=0, kolPerem=N, kudaPerem=<cell>`.

**Про ячейки** (1_CONST/02 § 5b): ячейки физически уже существуют на складе (приклеены к стеллажу с адресом), сотрудник их не создаёт. На листе `🍬 КОРОБЫ` ячейки могут отсутствовать, если в них раньше ничего не клали — тогда создаём как обычный короб с тарой `ЯЧ`.

**Sync flush**:

| Лист | Действие |
|---|---|
| `🍬 КОРОБЫ` `<src>` | `I -= N` |
| `🍬 КОРОБЫ` `<cell>` | если строка `(cell, barcode)` есть → `I += N`. Иначе append: `C=ЯЧ` (всегда, не вычисляется!), `D=<cell>, E=В СБОРКЕ, J=<адрес ячейки>, копии S/T/U из src`. На финише заявки `E:=ХРАНЕНИЕ`. |

**Тарификация источника**: paid (т.к. `units_to_cell > 0`).

## D. Изъятие в короб отгрузки (микс) `partial pick`

**Trigger**: модалка, для одного/нескольких баркодов `kolPodb>0, kudaPodb=<ship>` (короб отгрузки уже создан в A).
**Атом**: `box.set_layout` со смешанной раскладкой.
**Sync flush**:

| Лист | Действие |
|---|---|
| `🍬 КОРОБЫ` `<src>` | `I -= (kolPodb + kolPerem)` для каждого баркода |
| `🍬 КОРОБЫ` `<ship>` | если строка `(ship, barcode)` есть → `I += kolPodb`. Иначе append: `B=сегодня, C=<ταra-коэф>, D=<ship>, E=В СБОРКЕ, F=<zay>, G=УТ ГОТОВ, H=SKU(src), I=kolPodb, M=<склад>, N=<дата>, R=<габариты>\n<КЛ/ФФ>, S=МП(src), T=клиент(src), U=barcode, X=<zay>` |
| `🍬 КОРОБЫ` `<cell>` | как в C (если был kolPerem) |

**Тарификация источника**: paid если `units_to_cell > 0` ИЛИ `units_remaining > 0`.

## E. Микро-инвент `box.inventory_correction`

**Trigger**: тап на «Остаток» строки → numpad → новое `КОЛ`.
**Атом**: `box.inventory_correction { boxId, barcode, novKol, oldKol, reason }`.
**Sync flush**: `🍬 КОРОБЫ` `<src>` `I := novKol`.
**Тарификация**: **не влияет**.

## F. Финал заявки `zayavka.finish`

**Trigger**: кнопка «Завершить» в UI.

**Pre-check на фронте**:
1. По каждому баркоду заявки: `НУЖН - СОБР = 0` для **всех**? → автоматически полное завершение, без модалки.
2. Иначе (есть несоответствие — собрано меньше или больше) → модалка:
   - **«Полное завершение»** → СОБРАНО + блокировка на всех планшетах.
   - **«Частичное завершение»** → ЧАСТ.СОБР + другие планшеты могут продолжить.

**Атом**: `zayavka.finish { zayavkaNumber, mode: 'full' | 'partial' }`.

**Запись (forced flush, не через 2-мин очередь):**

При `mode='full'`:

| Лист | Действие |
|---|---|
| `🍬 КОРОБЫ` | для всех строк с `F=<zayavka> AND E=В СБОРКЕ` → `E := СОБРАНО`. **Исключение**: строки с тарой `ЯЧ` (ячейки) → `E := ХРАНЕНИЕ` (см. C). |
| `БД` строка `F=<zayavka>` | `U := СОБРАНО, V := now` |
| `ВР` | **в новой web-системе НЕ ИСПОЛЬЗУЕТСЯ** (отложено) |
| `НАЧИСЛЕНИЯ.НАЧ` | **отложено** — TODO: разобрать формулы из листа `РАСЧЕТ` (планшет подборщика); см. todo «разбор логики начислений» |

При `mode='partial'`:

| Лист | Действие |
|---|---|
| `🍬 КОРОБЫ` | строки `В СБОРКЕ` остаются как есть (можно вернуться) |
| `БД` строка | `U := ЧАСТ.СОБР, V := now` |
| `ВР` / `НАЧИСЛЕНИЯ` | **не пишем** (см. выше) |

**Тарификация (per короб-источник)**:
```
если units_to_cell(B) == 0 AND units_remaining(B) == 0 AND units_to_ship(B) > 0:
  charge(B) = 0   # free
иначе:
  charge(B) = units_to_ship(B) × 10₽   # paid (RATE_PER_UNIT)
total_charge = Σ charge(B)
```

**Контр-интуитив**: если из короба ушёл целиком ОДИН баркод, а другой остался / в ячейке — короб **paid** (он был открыт).

**Блокировка после finish:**
- При `mode='full'`: статус в БД := СОБРАНО. Все frontend'ы при следующем polling/refresh видят это и **блокируют** UI правок (кнопка «Заблокировано — заявка завершена»).
- При `mode='partial'`: статус := ЧАСТ.СОБР. Карточка в списке остаётся, кнопка «Продолжить →».

**Реализация блокировки на чужих планшетах**:
- Все правки sync POST'ятся на бэк. Бэк перед apply'ем атома проверяет в БД статус заявки. Если `СОБРАНО` → отклонить с reason `zayavka_completed`.
- Frontend показывает alert «Заявка уже завершена другим сотрудником» + блокирует UI.
- Polling /api/podbor/state теперь возвращает БД-статус заявки → frontend на основе него блокирует UI без необходимости отправлять атом.

## G. Откат заявки `zayavka.close` (если Σсобрано=0)

**Trigger**: «Закрыть без сборки».
**Атом**: `zayavka.close { zayavkaNumber }`.
**Запись:**
- `БД`: G := '', U := `СОЗДАНО`, V := now.
- Виртуальные коробы отгрузки (без материала) — удаляются из очереди.
- `🍬 КОРОБЫ`: ничего не пишем.

## Хранение состояния заявки (источник истины)

С 2026-05-11 — **source of truth = backend JSON-файл**, лист `🍬 КОРОБЫ` остаётся write-through кэшем для совместимости с downstream-системами.

### Структура

```
${PODBOR_DATA_DIR}/                  # абсолютный путь, ВНЕ директории релиза
  zayavki/<safe_id>.json             # state активной заявки
  _done/<safe_id>-<ts>.json          # архив после finish/close
  queues/<safe_id>.json              # буфер pending-операций к листу (sync-engine)
  locks/<safe_id>.json               # lock state
```

`PODBOR_DATA_DIR` env-переменная. Локально → fallback `${DATA_DIR}/podbor`. На VPS → `/var/lib/upseller-podbor` (чтобы не терялся при rsync-деплое).

### Содержимое `zayavki/<id>.json`

| Секция | Содержит |
|---|---|
| `meta` | client, mp, ks, warehouse, finalWarehouse, dateOtgr, status, pickers[], createdAt, startedAt, finishedAt, updatedAt |
| `request` | items[] из БД_ЭКСП.J (что нужно собрать) |
| `shipBoxes` | список созданных коробов отгрузки (number, short, tara, dimensions, owner) |
| `events` | append-only журнал всех действий: zayavka.start/finish, ship.create/delete, set_layout, full_to_ship, inventory_correction |
| `computed` | derived из events: pickedByBarcode, sourceBoxes (free/paid), nach (paidByBarcode, totalCharge) |

`events[]` — primary, `computed` — derived (всегда можно перезапустить пересчёт).

### Поток данных

```
Действие на фронте → POST /api/podbor/sync
   ├─ appendEvent в zayavki/<id>.json (atomic + per-zayavka mutex) ← primary
   └─ syncEngine.addOp в queues/<id>.json (batched flush к листу) ← write-through
```

При reload / disconnect / новый деплой:
- JSON на бэке хранит всё; восстановление бесшовное.
- Лист может отставать на ≤2 мин (batch tick), но при расхождении JSON прав.

### API endpoints

| Endpoint | Что |
|---|---|
| `GET /api/podbor/state?zayavkaId=...` | summary: pickedByBarcode, nach summary, shipBoxes, eventsCount, bdStatus |
| `GET /api/podbor/nach?zayavkaId=...` | детальный отчёт по начислениям: paid+free per barcode, totals |
| `GET /api/podbor/picklog?zayavkaId=...` | events заявки (timeline, последние сверху) |

### UI: кнопки в шапке полотна (после «▶ Начать»)

- **`💰 НАЧ · {sum}₽`** — live-сумма начислений, клик открывает модалку с таблицей paid+free per баркод.
- **`📋 ЛОГ · {N}`** — счётчик событий, клик открывает timeline всех действий.
- Polling каждые 10 сек обновляет цифры на кнопках без перезагрузки.

### Финиш заявки (mode='full')

```
1. flushQueue к листу (всё успело уйти).
2. finishZayavkaFull — массовый перевод В СБОРКЕ → СОБРАНО (ЯЧ → ХРАНЕНИЕ).
3. writeNachToSheet — append paid-баркодов в лист НАЧ.
4. markFinished — БД.U = СОБРАНО.
5. archive — перенос zayavki/<id>.json → _done/<id>-<ts>.json.
```

Если шаг 2/3/4 упал — НЕ архивируем, статус не СОБРАНО, заявка остаётся «В РАБОТЕ»/«ЧАСТ.СОБР» и финиш можно повторить.

## Решено / открыто

| Вопрос | Решение |
|---|---|
| Деление начислений между N сборщиками | **Не делим**. Начисления per-заявка, per-баркод × 10₽. |
| Запись в `ВР` при finish | **Не пишем** в новой системе (отложено). |
| Запись в `НАЧИСЛЕНИЯ.НАЧ` при finish | **Отложено** — нужен разбор формул из `РАСЧЕТ` (планшет подборщика). См. todo. |
| Атрибуты ячейки при первой записи | Тара `ЯЧ` всегда, статус `В СБОРКЕ`→`ХРАНЕНИЕ` на финише, остальные копии из источника. |
| Округление коэф тары | До **2 значимых цифр** (`К_0.5`, `К_0.51`, `К_1.0`, `К_1.25`...), не до жёсткого списка. |
| Источник после полного изъятия | `КОЛ` остаётся (не зануляется), статус `В СБОРКЕ`→`СОБРАНО` на финише. См. B (трансформация). |
