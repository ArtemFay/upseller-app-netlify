# Sobitiya — модуль «Журнал событий» (Event Journal)

Третий рабочий инструмент единого web-приложения UpSeller, рядом с `Podbor/` и `Invent/`. Фиксирует **каждое** изменение состояния системы (товары, коробы, заявки, начисления, права, статусы) — кто, когда, что, почему — и позволяет восстановить полную историю любого объекта.

> **Статус (на 2026-05-04)**: модуль на этапе проектирования. Реализация начнётся **после** закрытия `Podbor/SYNC_BACKEND_PLAN.md`. Сначала Подбор должен физически писать в UPSELLER, потом включаем event-emission, потом строим UI журнала.

## Назначение

Без журнала невозможно ответить на вопросы, которые в реальном WMS возникают каждый день:

- *«Кто и когда списал 12 шт товара X из короба K10234?»*
- *«Восстановите состав короба S1294-001 на момент 14:30 вчера»*
- *«Покажите всё, что делал Иванов за смену»*
- *«Когда у заявки S1340 статус сменился с СОЗДАНО на ЧАСТИЧНО СОБРАНА?»*

Журнал — единая append-only лента событий, общая для **всех** модулей: Подбор, Инвент, в будущем — Приёмка, Перемаркировка, Финучёт. Принцип: **изменение в системе без записи события — невалидно**.

## Технические принципы (детали — в CONTEXT.md)

- **Append-only**: события **никогда** не редактируются и не удаляются. Исправление = новое событие типа `correction`.
- **Идемпотентность**: каждое событие несёт `clientOpId` (UUID); повторная эмиссия с тем же `clientOpId` не дублируется.
- **Реверсивность через replay**: текущее состояние любого объекта можно построить как fold над его событиями.
- **Двухуровневое хранение**: hot-store (last 30 дней, fast UI) + cold-store (архив, всё за всё время).
- **Source-of-truth для аудита**: при расхождении журнала и Sheets — журнал прав, Sheets подлежит сверке.
