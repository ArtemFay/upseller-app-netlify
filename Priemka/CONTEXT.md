# PRIEMKA — контекст модуля

`PRIEMKA` — модуль приемки товара для единого сайта Upseller. Он создается вместо Google Apps Script-версии "Приемка 2.0", потому что GAS медленный, неудобен для TSD/планшетов и плохо подходит для дальнейшей серверной интеграции.

## Что строим

Новая приемка должна работать как web-приложение:

- на desktop 4:3 для старшего смены или оператора;
- на планшете 11" как основной экран приемщика;
- на телефоне как запасной компактный режим;
- на TSD Urovo T40 как scanner-first интерфейс с маленьким экраном.

Первичный UX-принцип: не копировать Google Apps Script буквально, а взять оттуда структуру данных и проверенную черновую логику, после чего собрать процесс по fulfillment best practice.

## Рабочий процесс

Базовый целевой процесс описан в [`docs/business-process.md`](docs/business-process.md). Коротко:

1. Оператор открывает приемку без предвыбранной поставки: список активных поставок загружается в фоне из Sheets.
2. Оператор выбирает поставку; только после этого загружаются товары заявки и создаются 9 закрытых пустых коробов с габаритами по умолчанию 60×40×40 см.
3. Приемщик открывает один короб сканом QR или тапом. Открытый короб становится активным; остальные короба остаются закрытыми.
4. Товар добавляется в активный короб сканом по 1 штуке или вручную указанным количеством.
5. Повторный скан QR активного короба или кнопка закрытия закрывают короб и освобождают выбор следующего короба.
6. Каждое действие фиксируется как событие: товар, короб, количество, сотрудник, устройство, время.
7. Система показывает расхождения, незакрытые короба, проблемные SKU и готовность к сохранению.
8. По завершении данные синхронизируются с основной таблицей/сервером.

## Источник данных

Актуальный GAS-анализ зафиксирован в [`docs/gas-v06-analysis-report.md`](docs/gas-v06-analysis-report.md). Важные вводные:

- таблица: `1wlz94rEXUEwkRLshk3l6YWXqMBBTSuClTWRU-Zbuvx8`;
- лист по умолчанию: `ПР`;
- поставка выбирается через `C4`;
- шапка приемки использует ячейки `C3`, `C8`, `C9`, `C11`, `C12`, `C13`, `C14`;
- строки товара читаются из диапазона `D:AQ`, начиная с 3 строки;
- короба лежат в колонках `N:AQ`, до 30 коробов на строку;
- GAS умеет черновики, выбор поставки, выбор тары, паллету, тип товара, доплату, комментарий, ручные позиции и печать этикеток.

## Где лежит код

- [`web/`](web/) — исходник текущего прототипа модуля.
- [`../web/PRIEMKA/`](../web/PRIEMKA/) — копия для публикации в общем сайте по URL `/PRIEMKA/`.
- [`docs/`](docs/) — документация процесса, правил разработки, GAS-анализа и плана миграции.

Пока нет build-step, изменения в `PRIEMKA/web/` нужно синхронизировать в `web/PRIEMKA/` перед локальным тестом или деплоем. Позже это стоит автоматизировать скриптом сборки.

## Правила дальнейшей разработки

- Новые рабочие файлы модуля добавлять в `Netlify/PRIEMKA`.
- Историческую папку `2_web/PRIEMKA` использовать как архив и источник анализа, не как место активной разработки.
- Для бизнес-решений сверяться с `docs/business-process.md` и обновлять его при каждом изменении процесса.
- Для правил ведения кода, README/CONTEXT и `1_CONST` сверяться с [`docs/development-rules.md`](docs/development-rules.md).
- Для интеграции с таблицами сверяться с `docs/gas-v06-analysis-report.md`.
- Для общего сайта учитывать корневой [`../CONTEXT.md`](../CONTEXT.md): авторизация, общий backend, deploy, соседние модули `Invent`, `Podbor`, `Sobitiya`.
- `1_CONST` обновлять, когда решение по приемке становится общесистемным знанием: новый статус, роль, источник данных, интеграция, событие, бизнес-инвариант или архитектурный принцип.
