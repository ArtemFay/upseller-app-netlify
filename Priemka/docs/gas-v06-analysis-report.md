## 2. Что уже есть в GAS v0.6

### Реальное чтение таблицы

`Code.js` уже не mock. Он читает данные из таблицы:

- `DEFAULT_SPREADSHEET_ID = 1wlz94rEXUEwkRLshk3l6YWXqMBBTSuClTWRU-Zbuvx8`;
- лист по умолчанию `ПР`;
- строки товаров читаются из диапазона `D:AQ`, начиная с 3 строки;
- баркод берется из `E`;
- габариты из `G:I`;
- вес из `K`;
- план из `M`;
- данные по коробам пишутся в `N:AQ`;
- на строку поддерживается 30 коробов.

Это критически важно для миграции: текущая таблица хранит распределение в парных строках, где верхняя строка содержит номера коробов, а нижняя строка содержит количества.

### Контекст запуска

`doGet(e)` принимает параметры:

- `spreadsheetId`;
- `sheetName`.

Это значит, что GAS-приложение уже умеет открываться не только на дефолтной таблице/листе, а в контексте конкретной таблицы.

Также есть функции:

- `buildLaunchUrl_`;
- `setLaunchLinkInPrA1`;
- `showLaunchDialogForActiveSheet`;
- `onOpen`.

То есть из Google Sheets создается ссылка “Открыть WMS приемку” в ячейке `A1`.

### Выпадающий выбор поставки

Добавлена серверная функция `getSupplyOptions(request)`.

Она читает варианты поставок из data validation ячейки `C4`. Если validation не удалось разобрать, есть fallback на лист `🔨`, диапазон `A2:A`.

Клиентская часть:

- `supplySelect`;
- `fetchSupplyOptions`;
- `switchSupplySelection`;
- `setSupplySelection`;
- polling `pollBootstrapForSupply`, чтобы дождаться пересчета листа после смены `C4`.

Это нужно обязательно перенести в будущую систему как отдельный сценарий выбора поставки.

### Форма приемки

В `HEADER_CELL_MAP` зафиксированы ячейки формы:

- дата: `C3`;
- приемщик: `C8`;
- тип товара: `C9`;
- чья тара: `C11`;
- паллеты: `C12`;
- доп. начисления/услуги: `C13`;
- комментарий: `C14`.

На фронте этому соответствуют:

- `formDate`;
- `receiverSelect`;
- `productTypeSelect`;
- `tareKlBtn` / `tareFfBtn`;
- `palletInput`;
- `extraChargeInput`;
- `commentInput`.

Списки приемщиков и типов товара берутся из data validation этих ячеек.

### Черновики

Добавлен скрытый лист `__WMS_DRAFTS`.

Функция `saveDraft(payload)`:

- сохраняет payload в скрытый лист;
- пишет дату/контекст/поставку/клиента/количество SKU/коробов;
- если JSON больше примерно 49k символов, сохраняет compact preview.

Также `readLatestDraftForm_` читает последнюю форму из черновика и объединяет ее с данными листа.

Это полезная концепция, но в web-версии вместо скрытого листа должен быть нормальный журнал/таблица событий в БД.

### Завершение приемки

Кнопка `Завершить` вызывает `saveCurrentDraft("finish")`.

Перед завершением клиент проверяет:

- заполнена дата;
- выбран приемщик;
- заполнен тип товара;
- выбрано `КЛ/ФФ`;
- заполнено количество паллет;
- созданы короба;
- хотя бы одна единица товара распределена по коробам.

На сервере при `reason === "finish"` вызывается `writeItemsAndBoxesToSheet_`, которая пишет распределение товаров по коробам обратно в лист.

### Короба

Короба создаются как:

`supplyCode + "-" + pad3(index)`

Например:

`P4796-001`

Есть:

- изменение количества коробов;
- добавление одного короба;
- габариты коробов `60×40×40 см`;
- запрет менять тару закрытого короба через массовое изменение;
- статусы open/closed;
- счетчик `Заполнено X из Y`;
- подсветка коробов, которые используются в строках/ручных строках.

### Товары

Для товара теперь учитываются:

- SKU;
- barcode;
- габариты;
- объем;
- вес;
- план;
- ручной факт;
- разложенное количество;
- ручной флаг `doneManual`.

Есть модальное окно редактирования габаритов товара.

Если габариты/объем/вес отсутствуют, UI помечает их как required/missing.

### Раскладка

Сохранились базовые сценарии:

- `Полностью`;
- `Частично`;
- `Авто по пустым`;
- `Завершено`;
- `Очистить`.

При `Авто по пустым` система сама создает недостающие короба, если пустых коробов не хватает.

Для `Частично` есть защита от повторного действия в течение 8 секунд.

### Ручные строки

Добавлена модель `manualRows`.

Это отдельная нижняя строка/строки, где можно выбрать баркод из каталога клиента или добавить новый товар:

- `manualRows`;
- `clientCatalog`;
- `makeManualRow`;
- `ensureManualRowsIntegrity`;
- `resolveItemByBarcode`;
- `ensureItemForBarcode`;
- `addManualItem`;
- `addAllClientItems`.

Это важная новая вводная: пользователь может не только работать по товарам из заявки, но и добавлять/подтягивать товары из каталога клиента.

### Поиск

В шапке есть поиск `headerSearch`.

`getVisibleItemEntries` фильтрует товары по SKU или баркоду.

### Сканер

Есть модальное окно `scannerDialog`:

- выбор активного короба;
- поле сканера;
- Enter добавляет +1 товар в выбранный короб.

Сканер сейчас работает по модели “выбранный короб в select + скан товара”. Это полезно, но для нашей новой системы нужно перестроить в LPN-процесс “сначала скан короба, потом скан товара”.

### Печать

Есть:

- печать коротких этикеток выбранного короба или всех коробов;
- печать состава короба;
- формат 100×150 мм;
- QR генерируется через `api.qrserver.com`;
- в QR кодируется `box.code`.

### UI

`Index.html` уже содержит:

- брендированную верхнюю панель UPSELLER;
- поиск;
- кнопки `Сканер`, `Печать`, `Состав`, `Завершить`;
- компактную боковую форму;
- список коробов;
- таблицу товаров;
- footer summary;
- модальные окна для короба, товара, сканера, частичного количества, сообщений, текстового ввода, габаритов.

Это хороший черновик desktop/tablet интерфейса, но еще не полноценный TSD-first процесс.

## 4. Что уже есть в нашем `receiving-web-app`, но иначе/лучше с точки зрения будущей системы

У нас уже заложены вещи, которых в GAS v0.6 нет как основного процесса:

- четыре режима интерфейса: desktop/tablet/phone/TSD;
- scanner-first режим для ТСД;
- требование сканировать короб перед товаром;
- запрет добавлять товар без активного короба;
- идея предварительной печати этикеток до наполнения;
- отдельная модель ошибок;
- общий журнал событий в UI;
- проект отделен от GAS и может стать нормальным web app.

Это оставляем как направление best practice.

## 5. Что в GAS v0.6 не соответствует целевому best practice

### Короб выбирается из select, а не сканируется как физическая единица

Для склада с несколькими приемщиками это риск. В новой системе короб должен становиться активным через скан QR/штрихкода на физической этикетке.

### Факт и распределение живут в клиентском state до сохранения

Если страница упала до `saveDraft`, часть действий может потеряться. В новой системе каждое действие сканирования должно отправляться на backend как событие.

### Нет idempotency key

Повтор запроса от ТСД или двойной Enter может удвоить количество. В backend нужна защита через `clientActionId`.

### QR через внешний сервис

Для production лучше генерировать QR локально или на backend.

### Нет реальной многопользовательской синхронизации

GAS v0.6 не дает live-состояния между несколькими устройствами. В новой системе нужны SSE/WebSocket или хотя бы быстрый polling.

### Табличная модель ограничивает процесс

Парные строки и 30 коробов на строку нужны для обратной совместимости с Google Sheets, но не должны стать внутренней моделью новой WMS. В новой БД источником факта должен быть `box_items`.

### Нет полноценного TSD UX

Модалка сканера полезна, но для Urovo нужен отдельный экран: активный короб, поле скана, последнее действие, крупная ошибка, минимум кнопок.

## 6. Предлагаемая стратегия миграции после анализа v0.6

### Шаг 1. Сделать в локальном проекте mock API в формате GAS v0.6

Формат bootstrap должен включать:

- `context`;
- `meta`;
- `form`;
- `formOptions`;
- `items`;
- `clientCatalog`;
- `defaults`;
- `supplyOptions`.

### Шаг 2. Добавить форму поставки в локальный прототип

Перенести поля:

- поставка;
- дата;
- приемщик;
- тип товара;
- КЛ/ФФ;
- паллеты;
- доп. услуги;
- комментарий.

### Шаг 3. Разделить доменную модель и Google Sheets adapter

Внутренняя модель:

- `supplies`;
- `supply_items`;
- `boxes`;
- `box_items`;
- `events`;

Adapter для старой таблицы:

- читает `ПР`;
- читает `C3/C4/C8/C9/C11/C12/C13/C14`;
- преобразует `D:AQ` в `items`;
- при необходимости пишет результат обратно в формат старой таблицы.

### Шаг 4. Перенести сценарии v0.6 в новую UX-логику

Перенести:

- поиск;
- ручное добавление товара;
- клиентский каталог;
- редактирование габаритов;
- done/manualFact;
- частично/полностью/авто;
- печать этикеток.

Но изменить:

- активный короб выбирать преимущественно сканом QR;
- товар добавлять как event;
- этикетка короба печатается до начала наполнения.

### Шаг 5. Backend

Когда frontend повторит возможности v0.6, заменить mock/GAS adapter на сервер:

- `GET /api/receiving/supplies/:id/bootstrap`;
- `POST /api/receiving/scans`;
- `POST /api/receiving/boxes`;
- `PATCH /api/receiving/boxes/:id`;
- `POST /api/receiving/print-jobs`;
- `POST /api/receiving/supplies/:id/finish`;
- `GET /api/receiving/supplies/:id/events`.

## 7. Вывод

GAS v0.6 стал не просто демкой, а важным источником требований:

- он показывает реальную структуру Google Sheets;
- содержит механизм выбора поставки;
- содержит форму приемки;
- умеет читать справочники из validation;
- умеет сохранять черновики и финально писать распределение обратно в лист;
- содержит много UX-мелочей для desktop/tablet.

Но как целевая система он остается табличным и однопользовательским. Для новой web-WMS его нужно использовать как adapter/spec source, а не как UX-эталон.
