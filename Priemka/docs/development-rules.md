# Правила разработки PRIEMKA

Дата: 2026-05-04

## Назначение

Этот документ фиксирует, как вести модуль `PRIEMKA` внутри единого сайта Upseller, чтобы при доработках не терять бизнес-логику приемки, связь со старой GAS-системой и общий контекст UpSeller.

## Где вносить изменения

1. Рабочий frontend-код новой приемки пишется в `web\PRIEMKA`.
2. Backend/API приемки пишется в `web\api\receiving-*.js` и `web\api\_lib\receiving`.
3. Историческая папка `2_web\PRIEMKA` не является местом активной разработки. Ее можно читать, копировать оттуда идеи и анализировать GAS, но не развивать как основную версию.
4. Модули `Netlify\Podbor` и `Netlify\Invent` не трогаются при задачах по приемке, кроме случаев явной интеграции, согласованной с пользователем.

Отдельной publish-копии больше нет: `web\PRIEMKA` является единственным источником правды и сразу отдается Express-сервером.

## Когда обновлять документацию модуля

Обновлять `Priemka\CONTEXT.md` и/или `docs\business-process.md` обязательно, если меняется:

- последовательность приемки;
- роль сотрудника или права действия;
- статус поставки, короба, товара или ошибки;
- правило сканирования, печати этикеток, закрытия короба;
- структура данных, API-контракт, импорт/экспорт в Sheets;
- UX-режимы для desktop, планшета, телефона, TSD;
- решение о том, что берем из GAS, а что заменяем на best practice.

`README.md` обновляется, если меняется способ запуска, структура папок, URL, статус модуля или краткое назначение.

## Когда обновлять `1_CONST`

`1_CONST` обновляется не при каждой мелкой правке интерфейса, а когда изменение влияет на всю систему UpSeller или будет важно в других сессиях:

- новое системное правило приемки или fulfillment-процесса;
- новый статус, роль, сущность, таблица, лист, колонка;
- новая интеграция между `PRIEMKA`, Sheets, backend, `Sobitiya`, WMS или VPS;
- изменение источника истины: например, факт теперь хранится в БД, а Sheets становится только внешним адаптером;
- решение, которое влияет на будущий `Podbor`, `Invent`, упаковку, ОТК, хранение или финучет.

Если правило относится только к локальному прототипу и не меняет бизнес-процесс, достаточно обновить документацию модуля.

## Технические принципы

- Backend-first: критичная бизнес-логика приемки должна переезжать на сервер, UI только помогает вводить и подсвечивать ошибки.
- Короб является центральной физической единицей учета. Товар добавляется в конкретный короб, а не просто в строку таблицы.
- Товар нельзя добавлять в короб без идентификатора и этикетки.
- Все критичные действия должны становиться событиями: скан товара, скан короба, создание/печать/закрытие короба, ручная правка, расхождение, завершение приемки.
- Для TSD каждое действие должно быть идемпотентным: повторный запрос не должен удваивать количество.
- Sheets/GAS на этапе миграции являются источником структуры и внешним адаптером, но целевая модель должна быть web/backend/DB.

## Проверка после правок

Минимальная проверка рабочего модуля:

```powershell
cd C:\Users\Psgl2\ai-projects\Fulfillment\2_web\1_UPSELLER_web
npm run dev
```

Проверить:

- `http://127.0.0.1:3010/PRIEMKA/` открывается на desktop;
- `http://<LAN-IP>:3010/PRIEMKA/` открывается с планшета в той же Wi-Fi сети;
- `http://127.0.0.1:3010/api/receiving/supplies` возвращает список поставок;
- `http://127.0.0.1:3010/api/receiving/bootstrap` возвращает bootstrap выбранной поставки;
- в консоли браузера нет ошибок JavaScript;
- основные действия приемки не ломают сценарии desktop/tablet/phone/TSD.

Статический запуск через `python -m http.server` больше не является полноценной проверкой: после добавления выбора поставки приемка зависит от API `/api/receiving/*`.
